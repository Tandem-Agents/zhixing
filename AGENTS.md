## 项目背景

知行(Zhixing)是一个可独立部署的个人智能体,运行在用户自己的设备上。

愿景是释放人类生产力:既做日常生活助手,也胜任任意工作场景。

## 模块索引

代码包(`packages/*`):

- `@zhixing/core` — 核心引擎:Agent Loop、事件系统、工具管线
- `@zhixing/orchestrator` — 编排器:Runtime 装配、子 Agent 生命周期、Task 工具
- `@zhixing/providers` — LLM Provider 层:协议适配、预设注册、配置解析
- `@zhixing/tools-builtin` — 内置工具:Read / Write / Edit / Glob / Grep / Bash
- `@zhixing/mcp` — MCP host:接入外部 MCP server 工具
- `@zhixing/network` — 网络出口原语:SSRF 安全 fetch、URL/IP 防护
- `@zhixing/server` — 常驻服务网关:HTTP + WebSocket + JSON-RPC 2.0
- `@zhixing/cli` — 命令行入口
- `@zhixing/channel-feishu` — 飞书通道(`packages/channels/`):长连接 + 卡片消息
- `@zhixing/test-utils` — 内部测试基础设施

研究与设计在 `research/`(`insights/`、`source-analysis/`、`landscape/`、`design/`)。其中:

- `research/postmortems/` — AI 协作失败复盘:记录多轮失败后才定位的问题、提炼可迁移诊断原则;调试前先扫其 README 反模式清单。

多 agent 协作面板(根级):

- `agent-board.md` — 单一工作目录下多 agent 并发开发时的共享活看板:原则区(进场开机自检) + 实例区(谁在跑) + 工作区(在改什么)。**并发协作时进场先读,按原则区登记自己、维护自己的块;**

## 行为准则(请严格执行)

### 1. 穷尽自力、基于事实,再找用户

任务尽最大可能自己完成:该查的信息主动查全——读源码、跑命令、搜资料,把问题基于事实彻底弄清,再判断是否真需要用户介入。只在确实卡死、或决策本身属于用户时,才带着已查清的事实去找他。

绝不允许:图省事跳过调查、用猜测代替事实、然后直接把问题丢给用户。

### 2. 直击核心,不啰嗦

回答前先抓准用户真正关心的需求点与核心问题,作答一针见血、命中要害。

不要:啰嗦铺垫、堆砌低相关信息、把重点淹没在废话里。

### 3. 改完代码先构建,再让用户测

用户用 `zz` / `zhixing` 运行的是构建产物 `dist/`,不是源码。改完代码必须先构建并确认成功,再请用户测——否则他测的是旧版本,白白污染整个验证回路。

改 CLI 用 `pnpm cli:build`;动到上游包(`core`、`orchestrator` 等)用 `pnpm build` 全量重建。

### 4. 动手需授权,提交是用户的事

只有用户明确授意才开始写或改代码;讨论、审视、设计阶段不擅自动代码。

只负责修改工作区文件;不得主动把文件放入暂存区(ADD 区),也不得把暂存区里的文件移出或改写暂存区状态。暂存、取消暂存、commit、push 都是用户的责任。

永远不自主 git commit / push:提交是用户的行为。即便被要求"整理成 commit",那也只是整理信息,不等于提交。

### 5. 先建可观测性,再精确定位

永远基于事实判断。遇到复杂问题,先想清楚怎么观测它——找到能亲自观测、验证真实问题的手段,再据此精确定位根因。

坚决抵制:多轮"猜 → 试 → 让用户验证",拿用户的时间当调试成本。
