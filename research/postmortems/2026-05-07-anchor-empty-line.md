# 2026-05-07 · AI 起首 ◆ 行视觉为空

## 问题现象

用户使用 zhixing CLI 时，AI 回复偶发出现 **◆ 锚行视觉为空**——锚字符 `◆` 单独占一行没有内容，下一行才是 hanging 4 缩进的实际文字：

```
  ◆

    你好！☀️

    现在是 19:52，需要帮忙吗？
```

预期：

```
  ◆ 你好！☀️

    现在是 19:52，需要帮忙吗？
```

特征：偶发（不是每次都触发）、token 计数正常（说明 LLM 在流式输出）、视觉上 ◆ 后什么都没有但下面内容是 hanging 续行格式。

## 失败诊断轨迹

### 第 1 轮：猜 zero-width 字符（ZWS / BOM / ZWJ）

- 用户截图给 AI 看
- AI 没有任何 LLM 真实输出的实证，凭"经验猜测"：可能是 LLM 输出了 zero-width 字符
- 修复：把 `LEADING_INVISIBLE` regex 从 `\s` 扩展到 `\s + U+200B-200D + U+FEFF`
- 让用户重启 cli + 重新触发 bug
- **失败**：用户复现，bug 仍然出现

### 第 2 轮：扩展到 Unicode `\p{Cf}` 类

- AI 推论：之前漏了某些 Unicode 格式控制字符，扩 trim 到 `\p{Cf}` 通用类一次性覆盖
- 修复：`LEADING_INVISIBLE = /^[\s\p{Cf}]+/u`
- 让用户重启 + 重试
- **失败**：用户复现，bug 又出现

### 第 3 轮：尝试"白名单"重设计

- AI 提议改方案：基于 `charWidth > 0` 找第一个可见字符（白名单代替黑名单）
- 用户**主动指出**："这个思路会不会让 ◆ 跳到第二行视觉上多一行间距？"——并问"我们这个修改 cli 独有还是 server 也有？"
- AI 检查后承认：白名单会漏 trim 起首空格（`charWidth(空格) = 1`）—— 自己提的方案有缺陷
- 退回 regex + 扩文档说明设计为"两类独立判断"
- **此时用户开始质疑**："为什么这个问题多次出现，是在修修补补吗？"

### 第 4 轮：用户主动建议加诊断日志

- 用户提议："先不解决问题，因为问题不明确，先加个办法让我看到每一次 AI 完整真实的输出"
- AI 这时才意识到：**之前所有诊断都缺真实数据，全是猜测**
- 实施：建 `llm-chunk-dump` 模块，环境变量 opt-in，把 LLM 每个 chunk 的 raw + codepoint hex 写到独立日志文件
- 用户开启 + 复现 bug 一次 + 把日志贴给 AI

### 第 5 轮：日志实证 → 一行修复

日志铁证：

```
[+26606ms] text_delta (len=1)
  raw: ""
  codepoints: U+007F     ← DEL 控制字符（C0/C1 区）
[+26658ms] text_delta (len=2)
  raw: "\n\n"
[+26720ms] text_delta (len=2)
  raw: "你好"
```

LLM（MiniMax）在某些 turn 起首发了**单独的 DEL chunk**（U+007F）。`LEADING_INVISIBLE` 当前是 `\s + \p{Cf}`，**DEL 在 `\p{Cc}` 类（控制字符）不在覆盖范围**——TextStream 写出 `"  ◆ <DEL>"`，DEL 不可见但占字符位 → ◆ 行视觉空。

修复：扩到 `\s + \p{Cc} + \p{Cf}` 覆盖完整不可见字符集合（与 `charWidth = 0` + 空白对齐）。一行 regex 改动 + 实证回归测试。

附带发现：用户截图 "1952" 没冒号也是 LLM bug——日志显示 LLM 在 `"20"` 和 `"17"` 之间发了 DEL 而非 `:`。这是 MiniMax 模型 BPE tokenizer 边缘 case，cli 端无法修复（LLM 没发 `:` 我们不能凭空补）。

## 用户成本量化

- 用户复现 bug 次数：**5+ 次**
- 其中**至少 4 次**是 AI 猜测假设让用户验证的成本
- 用户重启 zhixing 次数：≥ 5 次
- 用户的工作流被打断 N 次

如果 AI 第 1 轮就建议加诊断日志，**总成本会是：用户配合开 env var 一次 + 复现一次 + 给日志一次**——AI 看日志一次定位 → 一次修复。

## AI 的认知失败分析

### 失败 1：没有"先建可观测性，再定位"的反射

任何"用户看到 X，预期 Y"的视觉/渲染 bug，第一反应应该是"建立观测通道，让真实数据说话"。AI 的反应是"先 trace 代码 + 提假设 + 让用户验证"——把诊断成本转嫁给用户。

bug 类型识别错位：**视觉/渲染 bug 的真根因常在数据层（输入是什么），不在代码层（代码做了什么）**。代码 trace 只能验证"给定输入 → 输出"；真正的问题是"输入是什么我们不知道"——这类 bug **必须**有数据观测。

### 失败 2：猜测驱动的"扩范围 patch"模式

- 漏字符 → 扩字符列表
- 还漏 → 扩到 Unicode 通用类
- 每一轮看起来都"更通用"，但**没有任何一次基于实证**——都是被动响应症状

患在：扩 regex 字符类即使表面看是"加 Unicode 通用类"，本质仍是**猜测扩范围**。判断"修修补补 vs 根因修复"的标准不是"修法有多通用"，而是"是否基于实证"。

### 失败 3：没有觉知用户验证成本是稀缺资源

让用户做"重启 + 复现 + 验证"：
- 一次成本可接受（任何修复都需要终验）
- **超过一次同 bug 的复现验证就是浪费用户时间**——必须先自建诊断手段
- "我先试一下你试试"在用户角度等于"AI 在用我的时间做诊断"——anti-pattern

每多一轮"试试看"，用户对 AI 的信任度就下降一档。最终用户用"是不是修修补补"质疑，是已经临近信任崩溃的信号。

### 失败 4：过度自信于代码 trace

AI 反复 trace `markdown-stream` / `TextStream` 代码逻辑。code trace 是有用的工具，但有局限：
- 它只能验证"理论上代码对给定输入做什么"
- 它不能告诉我"实际输入是什么"
- 在数据层 bug 上，code trace 越多越深陷"我看代码逻辑没问题"的盲区

正确的反应：意识到"代码 trace 已经做了 30 分钟仍找不到根因" = 真根因不在代码层，**在输入层**。该停下 code trace，转去建立输入观测。

### 失败 5：用户先想到诊断方案是严重信号

第 3 轮用户主动提议"先不解决问题，加个办法让我看 AI 真实输出"——**用户比 AI 先想到要建立观测**。

这是 AI 协作中的**严重失败信号**：
- 用户来报 bug 是因为"用户不掌握工具但有现象"
- 诊断方法的设计**应该是 AI 主动**——这是 AI 的核心价值
- 当用户提出诊断方法时，等于用户已经替 AI 想了一步——AI 在该步上失职

应该把这个时刻视为"AI 立刻该做的事被用户先做了"的红灯。

## 提炼的原则

### 原则 1：观测优先（"Show me the data" 而非 "Let me think"）

当用户报告"屏幕显示 X，预期 Y"类 bug 且 AI 不能从代码立即定位时：

> 立即建立可观测性通道（日志 / dump / stderr trace），让真实数据告诉我们输入是什么；不要先 trace 代码 → 提假设 → 让用户验证。

时机判定：用户报 bug + AI 第一次代码 trace 没有立即定位 → **就该建观测**，不是再 trace 一遍代码。

### 原则 2：扩字符 / 扩规则的修复必须有实证

任何"扩 trim / parse / encode 范围"的修复：
- 必须有日志 / 测试输入 **证明漏的具体字符是什么**
- **没有实证就不要扩**——猜测扩范围 = 不可证明完整 = 必然再患

判断"修修补补 vs 根因修复"的硬标准：是否基于实证。

### 原则 3：用户验证成本是稀缺资源，且单调递减信任

让用户做"复现 + 验证"：
- 第 1 次：成本可接受（任何修复都要终验）
- 第 2 次：信号——AI 没找到根因
- 第 3 次：用户开始怀疑 AI 能力
- 第 4 次：用户应该（也确实）质疑"是不是修修补补"

超过 1 次同 bug 复现验证就是 anti-pattern。识别到"我已经让用户试过一次了"就该自建诊断，不是再猜一轮。

### 原则 4：视觉/渲染类 bug 默认在数据层

视觉/渲染 bug 的根因分布大致：
- **数据层**（输入字符串本身有什么）：~70%
- 渲染逻辑层（代码处理输入的算法）：~25%
- 终端 / 字体 / OS：~5%

视觉 bug 的第一反应应该是"先看数据"。code trace 是渲染逻辑层的工具，对数据层 bug 无效。

### 原则 5：诊断工具长期保留，opt-in 默认禁用

定位完一次后：
- 通用诊断工具（如 chunk-dump）**保留**作为长期可用工具
- 文档化 opt-in 接口（环境变量 / debug flag）
- 类比 cli 工具的 `GIT_TRACE` / `NODE_DEBUG` / `--verbose`
- 下次同类 bug 用户可直接 dump 给 AI 看，零额外开发成本

诊断工具不是"为本次 bug 临时建的"，是"为整类 bug 建的基础设施"。

### 原则 6：当用户先想到诊断方法时，立刻反思

用户主动提议建立观测 / 加日志 / 收集数据 = AI 在该步骤上失职的红灯。

立刻该做的事：

- 接受用户建议，立刻实施
- 当场承认"这一步本应该我先想到"
- 把这个时刻记入 postmortem，作为"AI 该做但没做"的具体例子

## 行动转化

为本项目沉淀的协作信号：

| 信号 | 解读 | 应该做的 |
|---|---|---|
| 用户同 bug 复现 ≥ 2 次仍未修好 | AI 在猜，没找到根因 | 停止猜测，建诊断观测 |
| 用户问"是不是修修补补" | 信任接近崩溃 | 立刻承认 + 转向根因路径 |
| 用户先想到加日志 | AI 在诊断设计上失职 | 立刻接受 + 当场反思 |
| 修复方案 = "扩字符列表" | 大概率是黑名单 patch | 必须有实证才能扩 |
| 视觉/渲染 bug + code trace 30 分钟无果 | 根因在数据层 | 转去建立输入观测 |

—

**本案最终修复**：1 行 regex（`/^[\s\p{Cc}\p{Cf}]+/u`）+ 实证回归测试 + 长期保留 `ZHIXING_RAW_DUMP` 诊断通道。
**理论最低成本路径**：用户配合 1 次复现 + AI 看 1 次日志 + 1 行修复。
**实际成本路径**：用户配合 5+ 次 + AI 4 次错误猜测 + 1 次根因修复。
