# Postmortems — AI 协作失败案例与反思

> **专放 AI 没能独立解决、多次失败后才解决的问题**——记录失败的诊断路径、暴露出的协作反模式、从中提炼的工作原则，让未来 AI 协作不重蹈覆辙。

## 这是什么

技术界标准的"事后剖析"（postmortem），但聚焦的不是产线事故，而是 **AI 协作过程中的诊断失败**——典型形态：

- 用户报告一个现象 → AI 提了若干假设 → 让用户验证 → 失败 → AI 提新假设 → 再验证 → 再失败
- 多轮"猜 → 试 → 失败"的循环把用户的时间用作 AI 的诊断成本
- 最终通过某个**结构性的认知转变**（不是又猜中了一个字符）找到根因

每个案例不是为了记录"这个 bug 怎么修的"——是为了暴露 **AI 在该案例中的认知盲区与反模式**，把"教训"提炼成可应用于未来同类场景的"原则"。

## 这不是什么

- ✗ bug 修复记录（那是 git history 的职责）
- ✗ 产品设计文档（那是 `design/specifications/`）
- ✗ 已对齐的产品决策归档（那是 `design/problems/`）
- ✗ 技术调研报告（那是 `landscape/` / `insights/`）

这里的内容是 **AI 协作元层面**的——关于"AI 怎么思考、什么时候该停下来观测、什么时候该承认猜测"。

## 何时新增一篇 postmortem

满足以下任一条件：

- AI 在同一 bug 上让用户配合验证 ≥ 2 次仍未解决
- AI 反复提"我先试一下" / "改一下试试看" 而非"先建立可观测性"
- 用户明确指出"是不是修修补补"或类似质疑，且事后证明确实是
- 通过加诊断工具 / 收集真实数据后才定位真根因

写作时间：解决问题后**立即写**，趁记忆清晰；不要拖到下个 bug 出现时才回忆。

## 文件命名

`YYYY-MM-DD-{短描述}.md`

日期前缀方便按时间排序；短描述用 kebab-case 让人一眼知道讲什么。

## 写作原则

写一篇 postmortem 时坚持：

1. **诚实记录失败轨迹**——不要美化"AI 经过深思熟虑后正确定位"。如果 AI 是猜的，就写"猜的"；如果用户先想到了诊断方法，就写"用户先想到的"。
2. **聚焦认知失败而非技术细节**——技术细节在代码 / 测试里。这里关心的是"AI 当时在想什么、漏了哪个角度、为什么没意识到"。
3. **提炼可迁移的原则**——案例是个例，原则是通用。每篇至少给一条"未来碰到同类信号时该做什么"。
4. **用户视角的成本量化**——记录用户为这次诊断花了几次复现、几次重启、多长时间。让"AI 用户视角验证成本"这个成本变量被显式化。
5. **避免引用易变标识符**——不要引用 Phase-N / Step-N / commit hash 等会变化的 anchor，让文档自身长期可读。
