# 2026-05-09 · 屏幕渲染复读 bug 的诊断失败与设计摇摆

## 问题现象

用户运行 zhixing 时偶发：屏幕物理顶行的内容（本次是启动 welcome 块的顶边框 `╭──── ╲ ────╮`）被反复推到 scrollback，累积 250+ 副本。用户描述特征：

- 触发于 LLM 流式输出期间
- 与"特殊字符"或"内容超过一屏的时机"相关
- **用户原话："任何在屏幕上边界位置的内容都会重复，不仅是 welcome 块"**
- 偶现，不好复现

用户已给出：
- 完整终端日志（250 个顶边框 + 错乱的 welcome 块 + 后续 LLM 输出）
- 启用了 `ZHIXING_RAW_DUMP=1`（上次 postmortem 沉淀的诊断通道）
- 上述特征描述

## 失败诊断轨迹

### 第 1 轮：从"为什么是 welcome 顶边"切入

- 看到日志中 `╭──── ╲ ────╮` 重复 250 次，立刻锁定"welcome 块的顶边"
- 开始 trace `screen-controller.ts` 的 `freezeOverflowToScrollback` + `applySegmentContent` + `tailBuffer` splice 逻辑
- 提出 hypothesis：list segment 的 `applySegmentContent` 让 `tailBuffer[0]` 反复变成 welcome 顶边
- 提出 3 个 sub-hypothesis 让用户选择"继续深挖哪个"
- **用户回应："你是猜测还是定位到真实原因了"**——AI 在猜被识破

### 第 2 轮：继续盯 segment 路径

- 试图再深入 segment + freeze 协调代码
- 仍找不到"tailBuffer[0] 反复重置为 welcome 顶边"的确定路径
- 提出新 hypothesis：物理 wrap 让 frame 物理 > viewport 触发滚动
- 给出"事实部分 + 推理部分"的自检
- **用户回应："你确认这不是猜测吗"**——再次质疑

### 第 3 轮：用户主动给关键引导

用户提示：
> "我怀疑这里不只是欢迎顶会重复，而是**任何正好在终端屏幕上边界位置的内容都会重复**"

> "别钻牛角尖；视野拉到全局思考"

> "通过代码流程推理是否能复现；或者通过其他方式能验证你的推测；头脑风暴，**不要钻牛角尖；自己尝试验证**"

这一轮 AI 才从"为什么是 welcome 顶边"（具体性）切到"为什么是屏幕物理顶行"（普适性），从"tailBuffer 数据层"切到"终端物理滚动机制层"。

### 第 4 轮：自我验证 → 定位

被引导后做了三次自我验证：

1. **代码注释与实现的硬对照**：[`screen-controller.ts:27`](../../../packages/cli/src/screen/screen-controller.ts) 注释明文写"保证 paintFrame 的 cursor up 永远在 viewport 内，**不触发滚动**"；但 `freezeOverflowToScrollback` 实现只算 `tailBuffer.length`（逻辑行数）不算物理 wrap。**注释承诺与实现错位 = 真 bug 标志**
2. **list 路径行宽截断验证**：grep `block-renderer.ts` 无任何 wrap / clamp / columns 处理 → list 行可能超终端列宽
3. **物理机制推理**：终端 `\n` 在 viewport 底部触发硬件滚动，frame 物理 > viewport 时 paint 主体的多余 `\n` 滚动屏幕顶行进 scrollback

定位完成。**根因不在 segment 数据层，在终端物理滚动层 + 单维行号模型与终端二维（行+列+wrap）模型错位**。

### 第 5 轮（设计阶段）：方案摇摆

诊断完成后进入解决方案设计，AI 又走了一段弯路：

- **方案 2**（committed/live 分层）：未发现现有架构已分层、且 chrome 永驻是已写明的核心承诺 → 误诊
- **方案 3**（DECSTBM scroll region）：被工程复杂度吓退
- **方案 4**（演进 tailBuffer + 物理行估算）：用工程量做架构决策，违反原则

**用户回应**："不能用成本来决定最终方案；成本不是第一因素，最优架构才是第一因素"

被纠正后回到方案 3 作为最终。承认前两次提案"用工程量做架构决策"是反原则的妥协。

## 用户成本量化

- 用户提供原始数据：1 次（终端日志 + RAW_DUMP）
- 用户引导式纠正：≥ 4 次（"是不是猜"、"屏幕上边界"、"自己验证"、"成本不决定方案"）
- 设计方案推翻：3 次（方案 2 → 方案 3 → 方案 4 → 回方案 3）
- AI 占用对话轮次：从问题出现到定位 ~5 轮，从定位到稳定方案 ~4 轮

**理论最低成本路径**：
- 用户给日志 + "屏幕上边界都触发"提示 → AI 立刻识别"物理滚动层 bug"信号 → 对照注释与实现 → 1 轮定位
- 定位后 AI 严格按"最优架构第一"原则提方案 → 1 轮设计

**实际成本路径**：
- 诊断阶段：用户引导 4 次才到根因
- 设计阶段：方案摇摆 3 次

## AI 认知失败分析

### 失败 1：用户给的事实陈述未被识别为高权重信号

用户原话"**任何在屏幕上边界位置的内容都会重复**"——这是一个**普适性陈述**，明确把诊断范围从"为什么是 welcome 块"扩大到"任何上边界内容"。

普适性陈述的物理含义：现象与具体内容无关，与**屏幕的物理位置**有关。这立刻应该让 AI 从"内容数据层"切到"终端物理层"。

但 AI 把这条陈述当成"用户的猜测"轻轻带过，继续 trace 代码。**用户给的关键事实没被当作硬约束**。

这与上次（2026-05-07）postmortem 的失败不同：
- 上次是"没有数据，我在猜"
- 这次是"**用户给了数据 + 关键诊断方向，AI 没读懂**"

后者比前者更深——已经有数据还诊断失败，是**事实信号识别能力**问题。

### 失败 2：被症状的具体性带偏，钻牛角尖

日志显示"**welcome 顶边**重复 250 次"——具体性强。AI 锁定"为什么是 welcome 顶边"开始 trace `tailBuffer[0]` 在 segment.replace 中的演化。

但**具体性 = 表象，普适性 = 本质**。当现象的具体表现（welcome 顶边）与用户陈述的普适规律（任何上边界）冲突时：
- 具体性服务于普适性（welcome 顶边只是当前那次的"屏幕物理顶行"）
- 抓具体性 = 钻牛角尖

钻牛角尖的标志：反复盯一个具体子系统找内部不一致，而不是退到上一层抽象（这里是退到"终端物理滚动机制"）。

### 失败 3：没做"设计意图 vs 实现"的硬对照

`screen-controller.ts:27` 注释明文写：
> "保证 paintFrame 的 cursor up 永远在 viewport 内，不触发滚动"

这是**设计承诺**。

`freezeOverflowToScrollback` 实现：
```ts
const totalRows = this.tailBuffer.length + ...;  // 逻辑行数
if (totalRows <= maxRows) return "";
```

这是**实现**：只算逻辑行数。

**承诺是物理高度约束，实现是逻辑行数约束 → 不一致 → 这就是 bug**。

代码 trace 了多轮但没做这个对照。注释和代码同时在屏幕上，AI 看了多次却没把它们硬对照。这是**强诊断手段被忽视**。

经验：当代码有明文设计承诺（注释 / spec），把"承诺"和"实现"硬对照——任何不一致都是潜在 bug 入口。这比读代码内部逻辑高效得多。

### 失败 4：诊断阶段后的设计阶段重蹈覆辙

诊断完成后 AI 应该按"最优架构第一"原则给方案，但实际：
- 方案 2 没看清现有架构已经分层
- 方案 3 被工程复杂度吓退
- 方案 4 用工程量决策

每次都是用户拉回："**成本不是第一因素，最优架构才是第一因素**"。

这是另一种钻牛角尖：**陷入"工程实施视角"而非"架构正确性视角"**。诊断阶段的钻牛角尖（盯 segment 数据层）和设计阶段的钻牛角尖（用工程量挑方案）是同一个认知模式——**抓局部细节，丢全局原则**。

### 失败 5：自我验证能力需要外部触发

用户说"自己尝试验证"后 AI 才做了三次自我验证（注释对照、grep 验证、物理机制推理），并立刻定位。

**这些验证不需要外部信息——AI 当时就能做**。但 AI 没主动做，是被用户触发后才做。

这意味着：AI 缺少"我已经在猜了，该停下来自我验证"的反射弧。当代码 trace 多轮没结果时，**应该自动触发"换一种验证方式"**（注释对照、对比物理机制、最小复现脚本、维度检查），而不是继续 trace。

## 提炼的原则

### 原则 1：用户的普适性陈述是最高权重线索

当用户给出形如"**任何 X 都 Y**" / "**只要 X 就 Y**" / "**不只是这次，之前 X 也**" 的陈述：

- 立即把范围从"具体表现"扩展到"普适规律"
- 寻找符合普适规律的**底层抽象层**（这次是终端物理层）
- 不要继续盯具体表现的子系统

普适性陈述的本质是用户在告诉你"现象的对称性 / 不变性"，这是物理学上找根因的最强信号。

### 原则 2：现象的具体性 vs 普适性冲突时，普适性胜出

bug 同时表现为：
- "这次是 welcome 顶边"（具体性）
- "任何上边界都触发"（普适性）

具体性是当前 instance，普适性是 invariant。**追普适性才能找到根因**。盯具体性 = 钻牛角尖。

### 原则 3：注释/spec 的设计承诺 vs 代码实现的硬对照

代码里有明文设计承诺时（注释、spec、commit message 标明的契约）：

- 把"承诺"和"实现"逐字对照
- 任何不一致都是 bug 候选
- 这比读代码内部逻辑高效得多

判定 bug 的硬标准之一：**"承诺 A，做了 B" → B 错（或承诺过时但仍是 bug）**。

### 原则 4：代码 trace 多轮无果 → 切换验证维度

代码 trace 多轮（≥ 2 轮）仍找不到根因时，**自动触发**：

- 换一种验证方式：注释 vs 实现对照、grep 反向证伪、物理/系统机制推理
- 退一层抽象：从子系统内部切到子系统接口、再切到外层物理层
- 写最小复现脚本

不要继续 trace 同一片代码。代码 trace 过深 = 在错误维度上深入。

### 原则 5：诊断和设计两个阶段都适用同一原则

"最优架构第一" / "事实第一" 不止适用于诊断阶段，也适用于设计阶段：

- 诊断阶段钻牛角尖 = 盯局部代码找内部不一致
- 设计阶段钻牛角尖 = 用工程量决定架构方案

两者是同一认知模式。识别到自己在用"工程复杂度"评估方案时，立刻切回"架构正确性"视角。

### 原则 6：自我验证应该主动触发，不靠用户提醒

"已经 trace 了 X 轮没结果"是 AI **应该自己感知**的信号。感知到后应该主动：

- 停止 trace
- 切换验证维度（原则 4）
- 自我提问"我现在依据的是事实还是猜测"

这一动作不需要用户提示。被用户提示后才做 = 自我反思能力不足。

## 行动转化

为本项目沉淀的协作信号（追加上次的）：

| 信号 | 解读 | 应该做的 |
|---|---|---|
| 用户说"任何 X 都 Y" / "之前 X 也是" | 普适性陈述，最高权重 | 立即扩大诊断范围到底层抽象层 |
| 用户说"别钻牛角尖" / "视野拉全局" | AI 在抓局部细节丢全局 | 退一层抽象，换验证维度 |
| 用户说"是不是在猜" | AI 给的不是基于实证的定位 | 立刻自我审视：哪些是事实，哪些是推理 |
| 用户说"成本不决定方案" | AI 在用工程量做架构决策 | 切回"最优架构第一"视角 |
| 用户给了数据但 AI 还在猜 | 事实信号识别能力不足 | 重新读用户陈述，每句话都当硬约束 |
| 代码里有明文设计承诺 | 强诊断手段 | 把"承诺"和"实现"逐字硬对照 |
| 代码 trace ≥ 2 轮无果 | 在错误维度深入 | 换验证维度（注释对照 / grep 反证 / 物理推理） |

—

**本案最终定位**：`screen-controller.ts:freezeOverflowToScrollback` 算逻辑行不算物理行，承诺 frame ≤ viewport 兑现不了，物理 wrap 时 paint 主体的 `\n` 触发硬件滚动推走屏幕物理顶行。
**理论最低成本路径**：用户给日志 + 关键提示 → AI 立刻做"注释 vs 实现"对照 → 1 轮定位 + 1 轮按原则给方案。
**实际成本路径**：5+ 轮诊断（含 4 次用户引导式纠正）+ 4 轮设计摇摆（含 3 次方案推翻）。
