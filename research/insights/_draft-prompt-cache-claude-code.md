## 1. 缓存基础（先理清原理）

### 工作机制

- 每次发请求，后台要把"系统指令 + 历史对话 + 新消息"从头到尾重新编码
- **提示词缓存 = 前缀匹配复用**：在请求里标断点，后台把"开头到断点之间"的编码结果存下来；下次前缀完全一样就直接复用，跳过重复计算
- **比喻**：每次写论文都要重抄目录和前几章，缓存让重复部分快进，只算新的那段

### 关键数字

| 项 | 值 |
|---|---|
| 命中缓存价格 | 0.1×（**省 90%**）|
| 首次写入价格 | 1.25×（**多 25%**）|
| 默认 TTL | 5 分钟，期间有请求**自动续期**不另收钱 |
| 付费 TTL 选项 | 1 小时 |
| 缓存最低门槛（多数模型）| 1024 tokens |
| 新模型门槛 | 4096 tokens |
| 短 prompt | 没有缓存的份 |

**实例**：10 万字长对话不开缓存每轮 0.30 美元；开缓存首次 0.375，之后每轮 0.03，**省约 90% 输入成本**。

### 缓存的额外收益

不光省钱——**延迟也降下来**。不用重算的部分越多，第一个 token 出来越快。

### Anthropic 内部的地位

把缓存命中率**当基础设施级别指标**——「跟服务器在线率差不多」：
- 命中率一掉触发值班告警
- 工程师按线上事故处理（原文「**宣布分级事故**」=「走完整事故响应流程」，**比普通告警严重多了**）
- 缓存命中率高的双重收益：
  - 不光省钱，**直接影响用户体验**
  - 让 Anthropic 能给**付费用户更宽松的使用额度**
  - 命中率越高，用户在同样价格下能用得越多

**核心立场**：「**缓存对 Claude Code 来说不是锦上添花的优化，是整个系统能跑起来的前提。没有缓存就没有 Claude Code。**」

理由：长对话「**一个会话几十轮**」，每轮都把上文带上重发；不复用就延迟和成本爆炸。

## 2. 七条经验

### 经验 1：提示词排列顺序——越不容易变的越往前

```
[最前]
  ① 系统指令 + 工具定义        所有会话共用、最稳定
  ② 项目文档                  同一项目内共享
  ③ 当前会话上下文             只在这一次对话里有效
  ④ 聊天消息（逐轮增长）       每轮只新增最后一条
[最后]
```

**比喻**：收拾书桌——常年不动的参考书放最底层，本周资料放中间，今天写的草稿放最上面。

### 经验 2：三个最容易踩的坑

| 坑 | 后果 |
|---|---|
| 系统指令里嵌**当前时间** | 每秒都在变 → 缓存直接废 |
| 工具定义用**无序容器**装 | 每次发请求顺序都不一样 → 前缀对不上 |
| **工具参数改了**（哪怕只动一个字段）| 整条前缀的缓存全失效 |

**核心**：一个小细节没注意，整条缓存链就断了。

### 经验 3：动态信息塞进消息流，不改系统指令

时间、文件状态等会变的信息怎么办？

- ❌ **不要去改系统指令**
- ✅ **把更新塞进下一轮的用户消息里**——附一段「系统提醒」，加载要更新的信息

**分法**：
- **系统指令** = 「地基」，盯死不动
- **消息** = 「流水」，想怎么改怎么改

这样前缀缓存稳。

### 经验 4：对话过程中不要切换模型——派子任务

直觉错的事：「简单问题切到小模型省钱、难题再切回大模型」。

**事实**：缓存**跟模型绑定**。换模型 = 之前积攒的所有缓存全部作废，从头重建。重建成本往往比让大模型直接回答简单问题还高。

**Claude Code 策略**：主对话**自始至终用同一个模型**。

需要小模型干活 → **派子任务**：
- 子任务有独立上下文和缓存——不污染主对话
- 主模型先写「任务交接说明」，把上下文浓缩好传给子任务
- 子任务做完只把结果传回主对话

**Claude Code 「探索模式」**就是这样工作的——小模型在独立上下文里执行探索任务。

**比喻**：你不会让实习生坐到你工位上用你的电脑。给他分配独立机器，写清楚任务发过去，做完发回来。

**附加事实**：缓存**按账号隔离**。
- 账号池中转：账号一混，**命中率过低，钱没赚到反而把号搞没了**（原文警示）
- 来回切账号：「别聊两句就来回切」——同样破坏命中率

### 经验 5：对话过程中工具集不要动

直觉错的事：「当前任务只需要 3 个工具，把另外 30 个移走更干净」。

**事实**：工具定义是**缓存前缀的一部分**。加一个减一个，整个对话的缓存全部重建。代价远超多放几个工具定义占的空间。

**Claude Code 「规划模式」**（进入后模型只思考、只规划、不执行操作）的实现：
- ❌ 直觉做法：进入规划模式时把执行类工具移走，退出再加回来
- ✅ Anthropic 做法：**保留所有工具不动**，加两个特殊工具：
  - `进入规划`（调用即切到思考模式）
  - `退出规划`（调用即回来）
- 「规划模式下不能执行」这个**约束通过对话消息传达**——在对话流里插一条系统消息「你现在在规划」

**关键区分**：
- **系统指令**是固定的、在缓存前缀里 → 别动
- **对话消息**是流动的、不影响前缀 → 想插什么插什么

**附加好处**：模型可以**自己判断什么时候进规划模式**，遇到复杂任务先想清楚再动手，不用用户手动切换。

### 经验 6：延迟加载工具

**问题**：Claude Code 要**接入几十个外部工具**——全部完整定义塞进去太占空间，按需加减又破坏缓存。

**Anthropic 折中方案**叫**延迟加载**：

- 一开始只放**轻量占位符**——模型看到的只是工具名字，不含完整参数定义
- 模型真要用某个工具了，再通过**「工具搜索」功能**去拉取完整定义
- **前缀始终只包含轻量占位符**，不会因为加载某个工具就变化

**比喻**：图书馆的书目索引——先翻目录找到想看的书，再去书架取，不用把所有书都搬到桌上。

**已开放**：这个「工具搜索」功能已通过接口对外开放，开发者可以直接用简化自己的工具管理。

### 经验 7：上下文压缩用「缓存安全分叉」

**问题**：长对话上下文窗口被填满，要把之前对话压缩成摘要。但如果另起一个请求做压缩——

- 用了不同的系统指令
- 没带工具定义

→ 从第一个字开始就跟主对话缓存对不上，**两条缓存链互相不复用**，且要把整段对话发过去做摘要，付完整无折扣的费用。**对话越长越贵**。

**Anthropic 解决方案：「缓存安全分叉」**

压缩请求必须用**跟主对话完全一样**的：
- 系统指令
- 用户上下文
- 工具定义

**步骤**：
1. 把主对话的消息作为历史**完整带上**
2. 在末尾**追加一条压缩指令**作为新的用户消息

**结果**：从后台视角看，这个请求和上一次几乎一模一样——**相同的前缀、相同的工具、相同的历史** → 前缀缓存可以**直接复用**。

**新增成本**：只有最后那条「压缩指令」本身。**一个压缩操作能复用主对话积攒的全部缓存，几乎不多花钱**。

**配套**：要**预留压缩缓冲区**——给摘要输出留个空间。**不能等窗口填满才开始压缩，要提前留余量**。

## 3. 元规则（七条经验背后的同一件事）

> 「**缓存就是前缀匹配。**」

| 元规则 | 表达 |
|---|---|
| ① 前缀匹配决定一切 | 前缀里**任何位置**的变化都会让其后所有内容的缓存失效 |
| ② 用消息代替指令修改 | 切换模式、更新时间——塞进对话消息，别动系统指令 |
| ③ 不要中途切换工具或模型 | 用工具调用表达状态转换；用延迟加载代替工具增删 |
| ④ 像监控在线率一样监控命中率 | 缓存中断 = 线上事故处理 |
| ⑤ 分叉操作必须共享主对话前缀 | 压缩、摘要、子任务全部用相同参数 |

**别改指令、别动工具——一碰整条缓存链就断**。换模型、切账号、另起炉灶，同一个道理。

---

## 4. 整体哲学

> 「这看着像缓存优化，但讲的也是一种思路：**先认死那条绕不开的限制，再围着它把整个系统搭起来。**」

Claude Code 的所有「设计选择」都是围绕「前缀匹配」这一约束展开的：
- 提示词分层、工具集稳定、状态用消息传、子任务独立、压缩前缀复用——

每个看似单点的设计决策，背后都是**「前缀不变」这条死线**。

## 5. 支线请求不会顶掉主线 cache

Prompt cache 是服务端缓存，不是"当前会话唯一槽位"，也不是模型内存状态。在 Anthropic 语义下，缓存命中看的是渲染后的 prompt 在 cache 断点前的精确前缀，前缀顺序是 `tools -> system -> messages`；在 DeepSeek / GLM / Qwen 隐式缓存语义下，provider 会自动识别公共前缀或重复上下文。共同点是：不同前缀会形成不同缓存条目，不会共用一个"当前槽位"。

同一时间可以存在多条缓存链：主线有主线的前缀，推进侧/支线有自己的前缀。支线请求命中或写入自己的 cache，不会把主线 cache 顶掉。回到主线时，只要主线请求的 `tools + system + messages` 前缀仍然一致，且缓存未过期、未被淘汰、路由仍能命中，就可以继续复用主线 cache。

但 TTL 是按缓存条目独立计时的：支线请求只会刷新它自己命中的条目，不会替主线条目续期。如果推进侧连续运行超过主线 cache 的有效期，而期间没有主线请求命中过那条前缀，下一次回到主线可能需要重新写入 cache。这不是"被支线顶掉"，而是"主线条目自己到期"。这个风险强依赖 provider：Anthropic / Qwen 显式缓存是分钟级；DeepSeek 是小时到天级磁盘缓存；GLM 官方未公开具体时长。

支线也有自己的 cache 经济性。是否有写入溢价取决于 provider：Anthropic 与 Qwen 显式缓存有首次写入溢价；DeepSeek、GLM、Qwen 隐式缓存没有公开的写入溢价，DeepSeek 缓存存储免费。因此在知行默认的隐式/自动缓存下，支线第一次运行不应默认按"写入惩罚"估算；但推进侧前缀稳定仍然有价值，因为它决定后续判断调用能否命中。

真正会破坏主线 cache 的，是主线请求自身发生变化：工具列表变了、system prompt 变了、历史消息前缀被改写、模型/provider/账号切换，或缓存 TTL/服务端淘汰导致条目不可用。失效是分层的：动 `tools` 是最顶层失效，会让 `tools + system + messages` 全部重建；动 `system` 会让 `system + messages` 重建；只动 `messages` 通常只影响消息层。

对知行的结论：推进侧可以作为独立请求运行；只要它不改写主线前缀、不把自己的思考塞进主线历史，就不会从机制上打断主 Agent 的 prompt cache。

### 5.1 多 provider 缓存机制对照（知行实跑模型）

| 维度 | Anthropic / Claude Code | DeepSeek | 智谱 GLM | 通义千问 Qwen |
|---|---|---|---|---|
| 命中依据 | 前缀内容 + cache_control 断点 | 自动识别相同前缀，磁盘缓存 | 自动识别重复/相似上下文 | 隐式公共前缀；显式 cache_control 可选 |
| 支线是否顶掉主线 | 否，独立缓存条目 | 否 | 否 | 否 |
| 是否手动标断点 | 支持/依赖 cache_control | 不需要、无手动断点 | 不需要、无手动断点 | 隐式不用；显式支持 cache_control，最多 4 个标记 |
| 命中折扣 | 约 0.1x | 约 0.1x | 约 0.5x | 隐式约 0.2x；显式约 0.1x |
| 写入溢价 | 有 | 无，存储免费 | 官方未公开写入加价 | 隐式无；显式 1.25x |
| TTL / 有效期 | 默认 5 分钟，可 1 小时 | 数小时到数天，不用即清 | 官方未公开具体时长 | 显式 5 分钟命中重置；隐式不定期清理 |
| 命中字段 | `cache_read_input_tokens` | `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens` | `prompt_tokens_details.cached_tokens` | `prompt_tokens_details.cached_tokens` |

可迁移结论：支线/推进侧不顶主线是通用结论；真正破坏缓存的仍是主线前缀自身变化（tools / system / history / model / account）。但 TTL、折扣、写入成本、最低门槛、是否能手动标断点都必须按实际 provider 官方文档判断，不能把 Anthropic 数字直接套到国产 provider。

## 7. 讨论引子（候选议题）

待你挑：

1. **CapabilityState 演化 vs 缓存命中率**——HOT_RETENTION_TURNS=7 的 LRU 降级让 tools[] 在长对话中**周期性变化**。每次工具进出都会破坏前缀缓存。这跟 Anthropic「工具集自始至终不动」是直接矛盾的。
2. **request_capabilities 是真延迟加载还是伪？**——Anthropic 的延迟加载是「占位符在前缀里恒定，真定义按需拉」；知行的 `request_capabilities` 激活后**真定义进入 tools[]**——下一次 LLM call 前缀变了。是否能改造为真占位符模式？
3. **LLMSummarizeStrategy 缓存安全性**——压缩请求是否复用主对话前缀？是否在「同 systemPrompt + 同 tools + 同历史 + 末尾追加压缩指令」的形态下运行？
4. **缓存命中率监控**——是否值得引入命中率指标 + 异常告警？
5. **`<turn-context>` 注入位置**——Anthropic 「动态信息塞下一轮消息」的做法已对齐，但当前实现把 turn-context 装进**最末尾 user message 的 text block 内部**（不是独立消息块）。这是优于还是劣于「独立 system reminder 消息」？
6. **多 provider 路由 vs 缓存绑账号**——secondary 角色 / flush LLM call 切到不同 provider/账号时，缓存语义？
7. **多视角并发 fan-out 的 cache 时机**——同一前缀的并发请求不会互相命中，因为缓存条目要等首个响应开始后才可读。多视角发散如果共享同一大前缀，先发一个请求并等它开始流式，再发后续并发请求，才能吃到同一前缀的 cache。

## 8. 经济视角 + attention 真实边界（决策依据）

### 8.1 cache 经济模型

继续旧对话第 N+1 轮 vs 新开对话第 1 轮（带摘要 M tokens），边际成本相等的临界点：

```
N* = [(p_miss / p_hit) × (S + M) - S] / (u + a)
```

**DeepSeek-V4-Pro 2.5 折价格**：`p_miss / p_hit = 3 / 0.025 = 120 倍`

| 场景 | S | u+a | M | **N\*** |
|---|---|---|---|---|
| 闲聊 | 15K | 300 | 5K | ~7950 轮 |
| 编程 | 15K | 3500 | 10K | ~853 轮 |
| 工具密集 | 25K | 2000 | 10K | ~2090 轮 |

**结论**：cache 经济上几乎不存在"主动切对话"的最优点——只要前缀稳定，继续累积永远比新开+摘要划算。

### 8.2 attention 才是先到的约束

主流国产模型有效上下文边界（基于 RULER / MRCR / NoLiMa / 业界基线）：

| 模型 | 总窗口 | 舒适区 | 警戒区 | 危险区 | 硬度 |
|---|---|---|---|---|---|
| DeepSeek-V4-Pro | 1M | ≤ 128K | 128K–512K（MRCR 82-94%）| > 512K（1M MRCR 0.59） | 🟢 官方 MRCR |
| DeepSeek-V4-Flash | 1M | ≤ 32K（保守）| 32K-128K | > 128K | 🔴 仅 1M 终点 |
| GLM-5.1 | 200K | ≤ 16K（RULER 96+）| 32K (93) → 64K (87) | 128K (78.86) | 🟡 DSA 消融基座 |
| Kimi K2.6 | 256K | ≤ 32K（前代代码场景）| 32K-128K | > 128K | 🟡 前代 K2 LongCodeEdit |
| Qwen-3.6（原生）| 262K | ≤ 32K（基线保守）| 32K-128K | > 128K | 🔴 官方仅 NIAH |

**业界基线**（RULER / NoLiMa）：标称 1M 实际可靠 50-65%；vanilla NIAH 已不可信；**NoLiMa 显示 11/12 模型在 32K 处掉到短文表现的 50% 以下**。

### 8.3 双约束并存的核心结论

| 约束 | 临界点 | 性质 |
|---|---|---|
| cache 经济 N* | 数百到数千轮（远）| 软约束 |
| attention 衰减 | **32K-128K**（近）| 硬约束 |

**用户实测交叉验证**：chat-20260504-41b4 出现注意力稀释症候时累积 ≈ 96K tokens——精确落在多数模型的"危险区入口"。

**对知行的直接含义**：
1. 上下文管理阈值必须按**绝对值（~32K / ~64K）**设，不是按总窗口百分比（v1.2 的 85% × 1M = 850K 与 attention 物理上限差 26 倍）
2. 在 attention 阈值之前，**cache 完整命中** > 任何形式的"主动管理 messages"——因为 cache 经济收益是确定的 120 倍，attention 改善是不确定的
3. 一旦逼近 attention 阈值，正确做法是经验 7「**缓存安全分叉**」式的整体压缩（同 system + 同 tools + 同历史 + 末尾追加压缩指令），而不是逐轮滑窗 / 逐轮锚化（后者每轮破坏 prefix）
