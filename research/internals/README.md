# 系统内情 (Internals)

> 知行自身系统的确定性工程事实:各技术域的能力边界、约束雷区、踩过的坑的根因与解法。做同类需求前先查这里。

## 定位

按技术域组织,只记「我们的系统现在的确定事实」。与邻居的边界:

- **vs `source-analysis/`** — 镜像:那边记**别人的**系统怎么实现,这里记**我们的**
- **vs `design/`** — 那边讲**未来要做什么**,这里讲**已经是什么**
- **vs `postmortems/`** — 见下
- **vs git history** — 那是修复流水,这里是沉淀下来的事实结论

## 与 postmortems 的分工

同一事件两副面孔,各记一面、互链:

| postmortems | internals |
|---|---|
| AI 的认知过程:怎么误判、提炼什么诊断原则 | 系统的事实结论:边界、验证方法、解法 |
| 面向「下次别这么想」 | 面向「下次直接查到答案」 |

## 结构

每个技术域一个子目录,域内渐进披露:

```
internals/
└── <domain>/
    ├── overview.md           承载模块 + 核心模型
    ├── capability-matrix.md  能力边界矩阵(多方案对比时)
    └── known-traps.md        雷区 + 根因档案(现象 → 验证 → 解法)
```

初期一域一篇 `overview.md`,长了再拆。每篇覆盖:承载文件、核心模型、能力边界与局限、约束雷区(带出处)、根因档案、关联指针。

## 维护原则

写 / 改本库文档守以下原则(都是踩坑提炼的,照做才出同质量):

1. **一手为准** — 断言落到一手源码 / 实测 / 官方源;**不拿本仓库的代码注释、设计文档当事实依据**(注释会过时、会引错号、会夹带立场)。
2. **分清事实与自辩** — 源码注释里的客观行为可采;「设计取舍的自我辩护 / 自我美化」是有立场的结论,不进事实库。
3. **守职责边界** — 只记本域(知行自身)的确定事实;竞品 → `source-analysis/`、未来设计 → `design/`、协作方法论 → `postmortems/`、修复流水 → git。越界内容删。
4. **不臆造、必核实** — 无一手依据不下因果断言;引 issue / PR 必核真实编号与内容;不确定的标「待核」或删,绝不冒充确定结论。
5. **不以偏概全** — 一个实例的做法别写成通用现状,先确认同类是否都如此。
6. **只留核心** — 去铺垫、反思口吻、某次需求的主观诉求、跨节重复;精简到只剩事实与出处。
7. **改完再审一轮** — 有修改就不算定稿,再独立审到「零发现、零修改」才收尾。

## 规约

- 域目录 / 文档名英文 kebab-case
- 确定结论、不可行项带出处(源码 / 实测 / issue),可复核可追溯
- 随实现 / 调研同步更新,不设审阅门禁
