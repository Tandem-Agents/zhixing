## 〇、定位与范围

Rubric 是推进侧使用的一等协议资产，与 Skill、Rule 同级但职责不同：

- **Skill**：教执行侧怎么做事。
- **Rule**：约束 Agent 不能 / 必须怎样。
- **Rubric**：交给推进侧判断任务是否完成、未完成如何续推、何时退出。

Rubric 不是执行方案，也不是运行中每轮协商的草稿。它是在任务第一次 run 前经用户确认的任务契约；一旦进入推进流程，推进侧按该 Rubric 自动验收和续推。

本文只定义 Rubric 作为协议资产的格式、字段语义、运行约束和校验要求。

- **本文范围**:Rubric 协议本身（身份字段、触发描述、content 结构、运行契约、校验清单、扩展原则）。
- **不在范围**:推进侧怎么创建任务闭环、怎么触发主 Agent、怎么持久化推进侧上下文、Rubric 库的具体 Store / 索引实现、用户确认交互细节。这些归属任务推进闭环架构或后续 Rubric 系统实现规格。
- **当前强制最小闭环**:只强制“通过标准”和“未通过时的处理”两块内容；退出边界、证据要求、优先级、变量定义、版本/来源等先作为扩展点保留。

## 一、协议目标

Rubric 当前先解决两件核心事：

1. **什么叫完成**：推进侧依据什么标准判断任务可以结束。
2. **没完成怎么推**：推进侧依据什么内容代替用户回复执行侧，触发下一轮 run。

协议必须开放、可扩展、可沉淀复用；但当前不追求一次性覆盖所有任务形态，先把最小闭环做稳。

## 二、基础结构

每个 Rubric 至少由三部分组成：

```text
title
description
content
```

落盘形态后置，但协议层先固定语义：

- `title` 是展示和引用名。
- `description` 是场景命中描述，决定推进侧什么时候应该用它。
- `content` 是推进侧实际执行验收和续推的契约正文。

### title

Rubric 的人类可读名称，用于用户确认、列表展示和引用。

要求：

- 简短、稳定、可复用。
- 表达场景，不表达某一次具体任务。

示例：

- `代码开发完成验收`
- `需求收敛推进`
- `文档审查完成验收`

### description

Rubric 的适用场景描述，用于推进侧按任务语义检索和命中。

要求：

- 写清楚“什么时候使用这份 Rubric”。
- 不写具体执行步骤。
- 可以包含反例，避免误命中。

### content

Rubric 的核心内容。当前至少包含两块：

1. **通过标准**
2. **未通过时的处理**

## 三、content: 通过标准

通过标准定义“什么情况下任务可以结束”。

要求：

- 面向验收，不面向执行。
- 尽量可观察、可核对、可引用证据。
- 可以是多个条件；推进侧必须按条件逐项判断。

结构建议：

```text
通过标准：
- <条件 1>
- <条件 2>
- <条件 3>
```

示例：

```text
通过标准：
- 用户提出的核心需求已经被明确回应或落地。
- 必要的文件修改已经完成，并能指出修改位置。
- 需要验证的场景已经验证，或明确说明无法验证的原因。
- 没有遗留与本任务目标直接冲突的问题。
```

## 四、content: 未通过时的处理

未通过时的处理定义“如果验收没过，推进侧应该怎么代替用户回复执行侧”。

这是用户参与的核心承载点：推进侧不能自由发挥，必须使用 Rubric 中用户确认过的推进内容。允许填入事实变量，但不能擅自改写推进意图。

要求：

- 写清楚未通过时给执行侧的回复内容。
- 回复内容是固定 / 非随机的任务契约内容。
- 可以包含占位变量，用于填充本轮事实。

结构建议：

```text
未通过时的处理：
- 场景：<未通过场景>
  回复：<给执行侧的固定推进内容，可带变量>
```

示例：

```text
未通过时的处理：
- 场景：仍有验收项未满足
  回复：当前任务还未达到验收标准。请继续处理以下未满足项：{missing_items}。完成后请给出对应证据。

- 场景：缺少验证证据
  回复：当前结果缺少可核对的验证证据。请补充验证过程、结果，或说明无法验证的具体原因。
```

## 五、运行契约

推进侧使用 Rubric 时必须遵守以下契约：

1. **先验收**:每轮 run 后先按“通过标准”逐项判断。
2. **通过即结束**:验收通过时结束推进流程，不再向执行侧发送代理回复。
3. **未通过才续推**:验收未通过时，匹配“未通过时的处理”中的场景。
4. **回复受限**:代理回复必须使用 Rubric 中用户确认过的固定 / 非随机推进内容。
5. **变量只填事实**:允许填入本轮事实变量，但不得借变量改写推进意图。
6. **缺口不自由发挥**:当前情况无法被 Rubric 覆盖时，不得擅自发明新推进策略；由任务推进闭环的退出 / 风险机制处理。

## 六、归属声明

Rubric 是可沉淀、可复用、可被用户长期维护的一等资产。具体 Store / 索引 / 检索实现不在本文展开，但归属必须满足：

- **资产管线**:Rubric 的落盘、读取、索引和用户管理应进入独立 Rubric 资产管线，不能混入普通对话历史。
- **渐进披露**:推进侧常驻上下文应只放 Rubric 索引或候选摘要；命中后再加载确认所需的 Rubric 正文，避免无界撑大上下文。
- **场景检索**:`description` 是命中依据的核心字段，应像 Skill 一样以“什么时候使用”为导向编写，而不是写成内容摘要。
- **安全与脱敏**:Rubric 是用户可长期复用、未来可能分享的资产；保存前应复用或对齐 Skill 资产管线的凭证脱敏、边界校验和来源标记纪律。
- **执行分工**:Rubric Store / 索引归 Rubric 系统实现；推进侧运行器只消费已确认 Rubric，不在运行中修改 Rubric。

## 七、扩展点

当前只强制 `title / description / content` 与 content 中的两块核心内容。后续可以在不破坏旧 Rubric 的前提下扩展：

- **退出边界**：死胡同、重复失败、风险、成本、需要用户裁决等停止条件。
- **证据要求**：推进侧验收时需要读取哪些产物、日志、测试结果或差异。
- **优先级**：多个未通过场景同时命中时的处理顺序。
- **变量定义**：`{missing_items}`、`{evidence}` 等占位变量的来源和格式。
- **版本信息**：Rubric 的版本、来源、最后确认时间、适用范围。

扩展原则：新增字段只能增强表达力，不能改变已确认 Rubric 的原有语义；旧 Rubric 在新协议下必须仍可解释。

## 八、校验清单

每份 Rubric 发布或保存前至少检查：

- 有 `title`，且标题表达场景，不表达某一次具体任务。
- 有 `description`，且描述“什么时候使用”，不是内容摘要。
- 有 `content`。
- `content` 中有“通过标准”。
- `content` 中有“未通过时的处理”。
- 每条通过标准面向验收，不写执行步骤。
- 每条未通过处理包含给执行侧的代理回复内容。
- 代理回复是固定 / 非随机内容；如有变量，变量只承载事实。
- 未把安全权限约束写成 Rubric 职责。
- 未把执行侧 Skill 内容混入 Rubric。

## 九、非目标

- Rubric 不教执行侧如何完成任务，那是 Skill 或执行侧策略的职责。
- Rubric 不承载安全和权限硬约束，那是 Rule 的职责。
- Rubric 不在每轮 run 后重新向用户确认；用户确认发生在第一次 run 前。
- Rubric 不允许推进侧在未授权情况下把固定推进内容改写成另一套推进策略。
