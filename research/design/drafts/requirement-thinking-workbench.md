## 维护原则

1. **回到需求本质**：先说清用户真实需要什么，再谈设计形态。
2. **避免架构反推需求**：不能因为已有底座，就反过来为它寻找产品理由。
3. **复杂性内收**：真正好的能力应让用户更容易完成生活和工作，而不是学习新概念。
4. **及时迁出**：需求收敛后迁入待做需求池、独立 draft 或 spec，本文件不长期堆历史。

## 当前问题

如何在保持 Skill 外形与心智不变的前提下，增加一层 runtime 可识别、可执行的分步编排协议，让 Skill 从一次性说明升级为可被系统分步投喂、分步执行、分步隔离的过程能力。

## 2. 临时草稿工作台梳理版

### 分步编排式 Skill（加强版 Skill 协议）

- **核心想法**：现有 Skill = 标题 + 描述（何时触发）+ 内容（怎么做），一次性把全部内容丢给模型、它自己干，只是"行动指引"、不参与编排。想做一个加强版：把一个 skill 的过程拆成**有限的 N 步**，内容**分步投喂、模型分步接收并执行**——做完第一步回到流程拿第二步，依次推进；最好做到步间隔离（第 k 步只看得见第 k 步内容，看不到前后步）。整体格式跟现有 skill 差别不大，核心是加一套**明确的分步标记 / 协议规范**，且步数必须有限。
- **为何更好**：原 skill 一次性给全、靠模型自律按文本走；分批给、分多次完成，单步注意力更聚焦，比"一次性丢全 + 自律"更稳。
- **关键约束**：这个协议**需要智能体内部编排配合**——现有 skill 平台无关、任何 agent 都能用；新协议要 runtime 真正实现"分步投喂 + 步间可见性隔离 + 推进控制"，所以先从知行自己的执行开始支持。
- **待定**：是否单独建一个仓库专门放这个协议（更专业、便于将来对外）。
- **与现有协议的关系（已查证）**：Anthropic 官方有 `github.com/anthropics/skills`（放官方示例 skill 内容）+ `agentskills.io`（Agent Skills 开放标准 / 协议本身）；其 SKILL.md 已有"渐进披露"，但那只是 **context 按需加载**（启动只读 name+description、激活才读全文、执行再按需加载附带文件），步骤仍写在文本里、靠模型自律执行。本想法的新意正是把步骤从"文本里的软建议"升格为 **runtime 强制编排的有限状态机 + 步间可见性隔离**——这正好对应"现有 skill 不参与编排"这个缺口。

## 3. 核心价值

核心价值不是"做一个更复杂的 Skill 格式"，而是把 Skill 从**静态知识包 / 操作说明**升级成一种**可被 runtime 执行的过程协议**。

现有 Skill 的价值是：告诉模型"遇到这类任务时，你应该参考这些方法、材料、脚本"。但它本质上还是一次性把说明塞进上下文，执行质量依赖模型自己记住、自己遵守、自己分阶段推进。

这个想法的核心价值是：把"怎么做"从模型自律，变成系统编排。也就是：

- 模型不是一次看到完整手册，而是每一步只拿到当前需要的指令。
- runtime 负责推进步骤、控制可见性、检查完成条件。
- Skill 不只是 prompt asset，而变成有限状态流程。
- 复杂任务可以被拆成稳定的、小上下文、高约束执行单元。
- 这会降低模型跑偏、偷步、漏步、过早架构化、一次性吞太多信息导致失焦的问题。

最有价值的一点是**步间可见性隔离**。这不是普通 workflow，也不是普通 checklist。它把上下文权限也纳入协议：模型在第 1 步时不应该被第 4 步污染，第 2 步也不应该提前看到答案形态。这个东西如果做成协议，会比"请你一步一步做"强很多。

所以它真正对应的方向可能是：**Runtime-Orchestrated Skill Protocol**，或者中文叫**编排式技能协议**。Skill 仍然是能力包，但其中一类 skill 可以声明为"需要分步执行"，由知行 runtime 按协议驱动。

## 4. 架构可行性与初步思路

架构上可行，但有一个关键分界：它不能只是 Markdown 约定，必须有 runtime 配合。纯靠文本写"第一步不要看第二步"做不到真正隔离；但如果由知行 runtime 负责解析协议、分步组装上下文、拦截未来步骤资源，就能做到"所见即所得"。

关键落点：

1. **外观看起来仍是 Skill**

   Agent Skills 规范本身就是 `SKILL.md` + frontmatter + Markdown body，且官方规范提供 `metadata` 字段给客户端放自定义属性，正文也没有格式限制。因此可以在普通 Skill 里加一个很轻的标记，而不是发明一个完全陌生的新文件类型。
2. **对普通 agent 是普通 Skill，对知行是可编排 Skill**

   示例：

   ```yaml
   ---
   name: deep-research
   description: Use when a task needs staged research and synthesis.
   metadata:
     zhixing.protocol: stepped-skill/v1
     zhixing.flow: workflow.yaml
   ---
   ```

   普通 agent 看到它，仍然是一个 skill；知行看到 `zhixing.protocol`，就知道不能一次性把全文塞给模型，而要进入"分步编排执行"。
3. **真正隔离靠 runtime，不靠模型自律**

   步骤内容不能全部写在 `SKILL.md` 正文里，否则 skill 一激活，模型就看全了。更合理的方式是：`SKILL.md` 只放触发说明、简介、兼容性说明；具体步骤放到 workflow / step 文件里，由 runtime 读取，但每次只注入当前 step。
4. **还要管住工具读取**

   如果 step 文件就在磁盘上，而模型可以用 Read 工具直接读 `steps/step-4.md`，隔离就破了。所以要么把分步内容作为 runtime 私有资源，不暴露成普通文件；要么在工具权限层做 step-scope 限制：当前只能读当前 step 允许的资源。

阶段判断：

> 编排式 Skill 是一种仍符合 Agent Skills 外形的技能包，但通过客户端可识别的协议标记，把部分说明从"模型一次性阅读"升级为"runtime 分步投喂、分步执行、分步隔离"的有限流程。

## 5. 补充目标与要求

### 5.1 协议专业度

协议必须顶级专业。要求不是"项目里能用就行"，而是至少要达到 Anthropic 官方 Skill 协议及其 GitHub 仓库的专业度，甚至更专业。格式、内容组织、说明方式、示例、兼容策略、版本策略都要有开放协议的水准。

### 5.2 完全兼容原有 Skill 协议

新协议必须支持原有 Skill 协议的全部内容。它不是另造一个新东西，而是在 Skill 基础上的增强 / 补充协议。一个编排式 Skill 首先仍然应该是一个合法、可被普通 agent 理解的 Skill。

### 5.3 价值与普及性

实现预期效果需要智能体侧支持，因此协议必须足够专业、足够有价值，才可能具备普及性或传播性。否则它会退化成知行项目独有的内部格式，单独建仓库和定义开放协议的意义就不大。

当前判断：市面上暂时没看到完全类似的方向，而这个方向确实有独立价值。初始目标不是只让知行能用，而是让别人也能使用；知行可以先作为贡献者和第一个实现者，把协议做专业。

### 5.4 智能体支持情况与预期效果

- **理想情况**：协议发展得好，智能体侧专门支持，runtime 能识别协议并执行分步编排、上下文隔离、步骤推进与完成判定。
- **早期情况**：智能体未支持时，把它当普通 Skill 处理。
- **目标**：协议架构足够完善时，支持它的智能体拿到该 Skill 后能完全发挥预期效果；即使没有智能体侧配合，至少也要比原来的 Skill 强。

### 5.5 核心需求与实现方式

更强的核心在于：能够**分布地让智能体读到步骤内容**。

- 如果要让智能体连续工作并严格推进，确实需要智能体编排；没有智能体配合，很难做到硬隔离。
- 需要继续探索是否存在更颠覆的方式：即使没有智能体主动配合，也能尽量达到"分步读取、分步执行"效果。

## 6. 关于"无智能体配合"的初步判断

结论先写清：**完全的"所见即所得"隔离，必须有执行环境配合；但无原生支持时，仍然可以做出比普通 Skill 更强的降级形态。**

### 6.1 不配合时做不到硬隔离

如果一个普通 agent 已经拿到了完整 `SKILL.md`，或者可以自由读取整个 skill 目录，那么它理论上就能看到后续步骤。此时协议文本再怎么写"不要提前看"，都只是软约束，不能构成真正的可见性隔离。

所以真正硬隔离至少需要其中之一：

- runtime 每次只注入当前 step；
- 工具层限制当前只能读取当前 step 允许的资源；
- step 内容不以普通可读文件暴露，而由受控 step provider 按状态返回。

### 6.2 可以做"渐进增强"的协议阶梯

协议可以设计成三层能力，而不是一刀切：

1. **普通 Skill 兼容层**：未支持协议的 agent 仍能读懂标题、描述、总体说明和保底执行方式。
2. **软分步层**：无原生支持但有 Read / Bash 等工具的 agent，可按说明一次只读取当前 step 文件，完成后再读取下一步；这不是硬隔离，但比普通 Skill 一次性塞全文更强。
3. **runtime 编排层**：支持协议的 agent 由 runtime 接管步骤推进和上下文注入，实现真正的分步投喂与步间隔离。

这条路让协议既像 Skill，又比 Skill 强：普通客户端不崩，愿意配合的客户端效果更好，原生支持的客户端获得完整能力。

### 6.3 可能的"颠覆性"方向

一个值得探索的方向是：把步骤内容设计成**受控资源**，而不是普通 Markdown 全文。

例如 `SKILL.md` 仍保持普通 Skill 外形，只声明：

- 这是一个 stepped skill；
- 普通 agent 的保底执行说明；
- step 入口或 manifest；
- 每一步完成后如何获取下一步。

具体 step 内容由一个 `next_step` 机制返回。这个机制可以是：

- runtime 内置 step provider；
- skill 包内的本地脚本；
- MCP server / tool；
- 远程或本地受控服务。

这种方式对普通 agent 的要求更低：它不需要理解完整协议，只要按 Skill 说明调用"获取下一步"入口即可。但它仍然不是零配合，因为 agent 至少要愿意按说明调用入口；硬隔离仍取决于未来步骤是否真的没有以普通文件形式暴露。

### 6.4 暂定设计原则

协议应避免只服务知行内部实现，而应内建降级语义：

> 不支持时是一个更清晰的普通 Skill；软配合时是一个渐进读取的 Skill；原生支持时才是完整的编排式 Skill。

这能同时满足三件事：保持 Skill 心智、争取传播性、给知行 runtime 留出真正实现隔离的空间。

## 7. 真正支撑的研究证据

本节只收录能支撑"固定子任务链式编排 / 执行中获取下一步信息"的证据。以下三类不作为核心论据：

- **长上下文退化 / context rot / lost in the middle**：只能说明"一次性塞太多信息可能不好"，不能直接证明"步骤内容链式获取"更好。
- **Agent Skills 的普通 progressive disclosure**：只能说明 Skill 已经接受"按需加载资源"的思路，但它主要是 metadata / SKILL.md / resources 的加载层级，不直接证明"步骤内容链式获取"更好。
- **耦合任务分解论文**：Least-to-Most、Plan-and-Solve 等证明"复杂推理可被分解并顺序求解"有价值，但它们处理的是一个耦合问题内的子问题依赖，不等于本需求的"多个相对解耦任务按协议串起来执行"。

### 7.1 场景校准

本需求不是"把一个本该整体处理的耦合任务硬拆开"。那类拆分可能丢失关键连续性，反而降低效果。

本需求的目标场景是：任务天然可以拆成多个相对解耦、顺序固定、边界清晰的子任务。用户今天只能手动把这些子任务一段一段交给 agent；stepped skill 想把这种人工链式交付变成可复用协议。

因此，真正需要支撑的是：

1. **固定子任务链式编排**在工程上有价值；
2. **模型能在执行过程中通过 action 获取下一段信息并继续**；
3. **协议自身是否更好**最终必须靠本项目 eval 验证。

### 7.2 直接工程依据：Prompt Chaining

- **来源**：Anthropic, [Building effective agents](https://www.anthropic.com/research/building-effective-agents)。
- **核心结论**：prompt chaining 适用于任务能被清晰拆成固定子任务的场景；它用更多延迟换更高准确率，因为每一次 LLM 调用都处理一个更简单、更聚焦的任务。
- **对本需求的支撑**：这是最贴近"解耦多任务编排"的官方工程背书。它支持的不是任意拆任务，而是**本来就能干净拆成固定子任务**的流程。
- **边界**：Anthropic 讨论的是多次 LLM call 的 workflow；stepped skill 目标是在一个 run 内通过静态链式读取推进步骤。二者执行形态不同，但"固定子任务链式编排能提高可控性 / 准确率"这一层高度相关。

### 7.3 直接机制依据：Action → Observation 循环

#### ReAct

- **论文**：Shunyu Yao et al., [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629), ICLR 2023.
- **核心结论**：让模型交替生成 reasoning traces 与 actions；actions 可以访问外部知识源或环境并返回 observation，模型再基于 observation 更新计划并继续。
- **对本需求的支撑**：stepped skill 的"获取下一步"正是一种 action → observation 循环：当前 step 执行完 → action 读取 next step → observation 注入下一段任务说明 → 继续执行。
- **边界**：ReAct 证明的是 reasoning + acting 的交替机制有效，不证明 stepped skill 的协议格式本身更优。

#### Self-Ask with Search

- **论文**：Ofir Press et al., [Measuring and Narrowing the Compositionality Gap in Language Models](https://arxiv.org/abs/2210.03350), Findings of EMNLP 2023.
- **核心结论**：self-ask 让模型显式提出 follow-up questions，并可接入搜索引擎获取后续答案，再继续回答原问题。
- **对本需求的支撑**：它证明了结构化提示可以驱动模型在过程中主动请求下一段信息，并把外部返回结果纳入后续推理。
- **边界**：Self-Ask 场景是组合式问答，子问题通常存在信息依赖，不是本需求的解耦多任务；它只作为"模型可按结构化协议请求后续信息"的机制旁证。

### 7.4 辅助背景：任务分解论文

以下论文不作为 stepped skill 的核心支撑，只保留为远端背景：

- **Least-to-Most Prompting**：Denny Zhou et al., [Least-to-Most Prompting Enables Complex Reasoning in Large Language Models](https://arxiv.org/abs/2205.10625), ICLR 2023。它证明复杂推理问题可通过子问题顺序求解改善泛化，但后续子问题依赖前面答案，且没有隐藏后续步骤内容，不证明链式按需获取步骤更好。
- **Plan-and-Solve Prompting**：Lei Wang et al., [Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning by Large Language Models](https://arxiv.org/abs/2305.04091), ACL 2023。它支持"先计划、再按计划执行"可以缓解漏步等问题，但没有"分布获取后续内容"这一点。
- **Decomposed Prompting**：Tushar Khot et al., [Decomposed Prompting: A Modular Approach for Solving Complex Tasks](https://arxiv.org/abs/2210.02406), ICLR 2023。它支持模块化子任务与不同 prompt / function 组合，但仍偏 prompt pipeline，不是 Skill 协议或静态链式读取机制的直接证据。

### 7.5 当前结论

这个需求的证据链应当这样摆：

1. **主梁**：Anthropic prompt chaining 支撑"干净拆分的固定子任务链式执行"有工程价值。
2. **机制支撑**：ReAct / Self-Ask 支撑"模型可以通过 action 请求下一段信息，并把 observation 接上继续"。
3. **远端背景**：Least-to-Most / Plan-and-Solve / Decomposed Prompting 只能说明任务分解、顺序执行、模块化 prompt 大方向有价值，不能直接证明 stepped skill。

stepped skill 的新意不是重新证明"分解任务"。它的真正命题是：

> 当一个 Skill 表达的能力天然包含多个固定、相对解耦的子任务时，把步骤内容组织成静态链式读取协议，让模型完成当前 step 后再获取下一 step，是否比普通 Skill 一次性给出全部说明更稳、更省心、更可复用。

外部证据能证明这个方向**合理且不荒唐**，但不能替代本协议自己的验证。仍需补的硬证据：针对同一批复杂真实任务，比较普通 Skill vs stepped skill 的成功率、漏步骤率、跑偏率、token 使用和人工修正次数。只有做出这个 eval，协议价值才能从"有机制与工程依据"升级为"本协议自身被验证"。
