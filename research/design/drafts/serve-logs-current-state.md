# `zz serve logs` 现状记录

> 临时文档：只记录当前已实现功能的事实、用途、使用范围和问题边界。后续如果要补齐日志治理能力，本文可过渡为架构设计文档；实现完成并沉淀后删除或归档。

## 保真结论

`zz serve logs` 是后台宿主日志查看命令，不是打开日志的开关。

它读取后台 daemon 宿主的持久日志文件：

```text
<ZHIXING_HOME>/server.log
```

默认情况下也就是：

```text
~/.zhixing/server.log
```

该文件由后台 daemon 启动时创建，并把 daemon child 的 stdout / stderr 追加写入同一个文件。当前实现没有看到针对 `server.log` 的大小上限、轮转、保留天数或自动清理机制。

因此当前 `server.log` 的治理状态是：只追加写入，依赖用户或外部系统手动删除 / 截断；产品自身不负责控制体积。

## 已实现功能定义

`zz serve logs` 的职责是查看后台宿主日志。

当前支持：

- `zz serve logs`：读取 `server.log`，输出最后 N 行。
- `zz serve logs --lines <n>`：指定输出行数，默认 50 行。
- `zz serve logs --tail`：持续追踪新增日志，轮询文件增长并输出增量内容。

它的功能边界是“读取后台宿主日志”，不是“配置日志系统”，也不是“开启诊断 dump”。

## 与 `zz --log` 的区别

`zz --log` 是当前 CLI 会话的诊断开关，用于打开 LLM raw chunk、keypress 路径等调试 dump，主要写入：

```text
<ZHIXING_HOME>/logs/
```

`zz serve logs` 是后台服务宿主日志的查看器，读取的是：

```text
<ZHIXING_HOME>/server.log
```

两者不是一类能力：

- `zz --log`：让当前 CLI 进程额外产生诊断日志。
- `zz serve logs`：查看已经存在的后台 daemon stdout / stderr 日志。

## 它有什么用

后台宿主是长期运行的服务进程。它不直接占用当前终端，用户遇到服务启动失败、后台服务异常、连接不上宿主、通道或调度相关问题时，需要一个低成本的观测入口。

`zz serve logs` 的价值是：

- 让用户或开发者看到后台宿主真实输出，而不是只看到 CLI 前台命令的抽象错误。
- 支持服务启动失败后的排障：启动流程本身也会提示日志路径。
- 支持 `--tail` 持续观察后台服务行为，方便定位偶发错误或异步通道问题。
- 为未来自有 App、飞书等接入面排障保留一个本机服务观测口。

## 给谁用

它不是普通用户的日常功能入口。

合理使用对象是：

- 开发者：定位后台宿主启动、运行、通道、调度、RPC 网关等问题。
- 高级用户或支持场景：按提示查看日志，协助判断本机服务状态。
- 未来自有 App / 第三方接入面排障链路：当接入面无法连接本机宿主时，可把后台宿主日志作为诊断材料。

## 当前问题

### 1. `server.log` 缺少尺寸治理

当前 daemon 启动时以追加模式打开 `server.log`，stdout / stderr 持续写入该文件。

没有看到以下能力：

- 文件大小上限。
- 日志轮转。
- 保留天数。
- 保留文件数。
- 启动期自动清理。
- 写入期自动截断或切分。

这意味着后台宿主如果长期运行，或某个异常路径持续输出日志，`server.log` 会持续增长，直到用户或外部系统手动处理。

### 2. `serve logs` 能感知外部截断，但不会主动治理

`--tail` 模式能识别文件被外部截断的情况，并从头重新读取新内容。

这只说明它能适应“别人已经截断了文件”，不代表系统自身具备日志截断、轮转或清理能力。

### 3. 现有日志清理不覆盖 `server.log`

当前已存在的日志裁剪逻辑主要面向 `~/.zhixing/logs/` 下的诊断 dump 子目录，例如 LLM raw / error 日志。

`server.log` 位于 `~/.zhixing/server.log`，不在这些子目录裁剪范围内。

## 目标效果

后台宿主日志应该同时满足两个目标：

1. **可观测**：服务异常时，用户、开发者和未来接入面仍能拿到足够的后台宿主日志。
2. **可治理**：日志不能无限增长；系统应有明确的大小上限、轮转策略、保留策略和自动清理机制。

目标产品体验是：

- 用户不需要理解日志文件生命周期，也不用担心 `server.log` 无限膨胀。
- `zz serve logs` 仍能查看最近有效日志。
- `zz serve logs --tail` 仍能追踪当前活跃日志。
- 后台宿主日志策略与 `zz --log` 诊断日志策略边界清晰，不互相混淆。
- 未来接入面需要诊断时，可以依赖一个受治理的后台宿主日志通道，而不是解析无上限的裸文件。

## 事实依据

- `packages/cli/src/index.ts` 注册 `serve logs` 子命令，并调用 `runLogsCommand`。
- `packages/cli/src/serve/logs.ts` 读取 `getDefaultLogPath()`，默认输出最近 50 行，`--tail` 通过轮询读取新增内容。

## 架构设计

### 设计判断

`server.log` 缺的不是一个查看命令，也不是一次性清理脚本，而是完整的生命周期治理。

后台宿主日志是本地服务的运行时资源。它的产品价值是“出问题时可观测”，但它不是用户资产、不是长期记忆、不是会话历史，也不应该无限占用用户磁盘。因此最优架构必须同时成立：

- 日志必须足够可靠，让后台服务异常时仍有材料可查。
- 日志必须有硬边界，不能依赖用户手动清理。
- 日志策略必须由系统统一定义，而不是 `serve logs`、daemon、未来 App、飞书通道各自处理。
- `zz serve logs` 只负责读取和展示，不负责决定写入、轮转、保留和删除。

### 核心定义

建立一个“后台宿主运行日志”能力，作为后台宿主运行时的一部分。

它包含三层：

- **日志策略**：定义路径、大小上限、轮转规则、保留规则、清理规则。
- **日志捕获层**：后台宿主通过 OS fd 级 stdout / stderr 重定向捕获运行输出，保证异常现场完整进入活跃日志。
- **生命周期管理器**：后台宿主启动后监控活跃日志文件，负责 copy-truncate 轮转和保留清理。
- **日志读取器**：`zz serve logs` 通过读取器查看当前日志和尾随新增内容，不直接承担生命周期治理。

这个边界很重要：生命周期治理必须靠近后台宿主运行时，因为只有运行时能持续观察日志增长；查看命令只能观察文件，不能作为系统治理的唯一入口。

### 目标路径模型

后台宿主日志应归入统一日志目录，而不是长期散落在 `<ZHIXING_HOME>` 根目录。

目标路径：

```text
<ZHIXING_HOME>/logs/server/server.log
```

轮转文件：

```text
<ZHIXING_HOME>/logs/server/server-YYYYMMDD-HHMMSS-SSS-<seq>.log
```

原因：

- `logs/` 是日志资源的统一边界，和 LLM / 诊断 dump 在物理层级上对齐。
- `server/` 是后台宿主日志自己的治理域，不和 `llm-raw`、`llm-error`、keypress dump 混在一起。
- `server.log` 保持为稳定的当前活跃文件，`zz serve logs` 和用户心智简单。
- 轮转文件用毫秒级时间戳加序号命名，避免同一秒多次轮转产生冲突，也避免跨平台 rename 链式操作复杂化。

如果实现时需要兼容旧的 `<ZHIXING_HOME>/server.log`，兼容只应发生在迁移边界：发现旧文件后迁移或读取一次，不能让新旧两个路径长期并存成双事实源。

### 默认生命周期策略

后台宿主日志必须有硬上限。默认策略建议：

- 活跃文件上限：8 MiB。
- 轮转文件最多保留：5 个。
- 轮转文件最长保留：14 天。
- server 日志目录总量上限：64 MiB。

执行规则：

- 后台宿主启动时检查活跃文件；如果超过上限，先轮转再继续运行。
- 后台宿主运行期间由生命周期管理器周期检查活跃文件大小；超过上限时执行轮转。
- 每次后台宿主启动时执行一次清理。
- 每次轮转完成后执行一次清理。
- 清理按“过期文件优先、超数量再删、超总量最后兜底”的顺序删除最旧轮转文件。
- 活跃文件永远保留；如果活跃文件超过上限，只能轮转，不能直接删除。

这些数字不是用户承诺，而是产品默认值。未来如果要暴露配置，也应通过统一配置模型进入日志策略，而不是给 `serve logs` 增加零散参数。

### 写入与捕获架构

后台宿主日志的第一优先级是完整捕获异常现场。

当前 OS fd 重定向模型有一个关键优点：daemon child 的 stdout / stderr 直接指向日志文件，因此 `console.*`、`process.stdout/stderr.write`、依赖库输出、Node warning、未捕获异常栈、硬崩溃时写到 fd 2 的内容，以及继承 stdio 的子进程输出，都能进入同一日志。

这个捕获完整性是产品价值的一部分，不能为了轮转而牺牲。

目标写入机制选择：

- 保留 OS fd 级 stdout / stderr 捕获。
- 父进程启动 daemon child 前，打开当前活跃日志文件并把 child stdout / stderr 重定向到该 fd。
- daemon child 启动早期初始化 `ServerLogLifecycle`，它不替代 stdout / stderr 捕获，只负责检查活跃日志大小、执行 copy-truncate 轮转、执行保留清理。
- 轮转时先把当前活跃文件复制为唯一命名的轮转文件，再对活跃文件执行 truncate，让已被 daemon 持有的 fd 继续写同一个路径。
- 不依赖 rename 一个打开中的活跃日志文件；这在 Windows 和长期持有 fd 的场景下都不应作为基础机制。
- 不把 `process.stdout/stderr.write` 或 `console.*` 拦截作为唯一捕获路径；进程内拦截抓不到 native 直写、硬崩溃输出和某些子进程输出，不能作为后台宿主日志的主路径。

显式 logger 仍然可以作为未来结构化日志入口，但它只是更好的日志生成方式，不是后台宿主日志捕获完整性的唯一来源。

copy-truncate 的取舍是明确的：它保留 fd 级完整捕获通道，并给活跃文件加上生命周期治理。后台宿主日志是诊断资源，不是审计账本；在本地小文件轮转的极短窗口里，理论上可能存在写入与复制 / 截断交错的边界风险，但这比永久丢失硬崩溃、native 输出和依赖库 stderr 更符合产品本质。

### 读取架构

`zz serve logs` 的职责保持简单：

- 默认读取当前活跃日志的最后 N 行。
- `--tail` 跟随当前活跃日志。
- 当活跃日志因轮转变小、消失或被替换时，读取器重新打开当前活跃路径。
- 读取器可以展示“日志已轮转”的提示，但不负责触发轮转。

`serve logs` 不应该：

- 删除日志。
- 截断日志。
- 决定保留策略。
- 解析日志语义并影响后台宿主行为。

它只是后台宿主日志的接入面投影。

### 与 `zz --log` 的关系

`zz --log` 和后台宿主运行日志必须继续分离。

- `zz --log` 是用户显式打开的当前 CLI 诊断 dump，目标是复现渲染、流式、按键输入等问题。
- 后台宿主运行日志是 daemon 的常规运行观测，默认存在，但必须受生命周期治理。

两者可以复用底层“日志目录裁剪”工具，但不能混用语义：

- `zz --log` 不能变成后台宿主日志开关。
- `zz serve logs` 不能查看 LLM raw chunk 或 keypress dump。
- 统一的是路径和治理原语，不是产品入口。

### 产品效果

用户侧理想效果：

- 后台服务出问题时，用户可以执行 `zz serve logs` 看到最近有效日志。
- 用户不需要理解日志轮转，也不用担心 `.zhixing` 被 `server.log` 无限撑大。
- 长期运行后台宿主不会制造无边界文件。
- 支持人员和开发者可以依赖一个稳定路径拿到最近服务现场。
- 未来自有 App 或第三方接入面需要诊断时，可以引用同一后台宿主日志能力，而不是发明各自的日志文件。

### 架构不变量

- 后台宿主日志是运行时观测资源，不是用户长期数据。
- 日志生命周期由写入端治理，读取端只展示。
- 活跃日志路径稳定，轮转文件只作为保留现场存在。
- 所有默认路径从 `ZHIXING_HOME` 派生，不能绕过统一路径模块。
- 所有大小上限按字节定义，不能按行数假装控制磁盘。
- 清理失败不能影响后台宿主启动，但必须写入可观测错误。
- 日志治理不能依赖用户主动运行某个命令。
- fd 级 stdout / stderr 捕获是后台宿主日志的基础能力，不能退化成只捕获显式 logger。
- 活跃日志 fd 必须以 append 模式打开；copy-truncate 依赖每次写入落到当前 EOF，避免 truncate 后继续按旧偏移写入造成稀疏文件。
- 轮转不能依赖 rename 一个已打开的活跃日志文件。
- 轮转文件名必须 collision-safe，同一秒内多次轮转也不能覆盖旧日志。
- 进程内 `console.*` / `process.stdout/stderr.write` 拦截不能作为唯一捕获路径。

### 验收标准

- 后台宿主运行过程中，`server.log` 不会无限增长。
- 活跃日志超过大小上限后会生成轮转文件，并重新写入新的活跃日志。
- 超过保留数量、保留天数或目录总量上限的旧日志会被自动删除。
- daemon child 的 `console.log`、`process.stderr.write`、Node warning、未捕获异常栈进入后台宿主日志。
- 继承 stdout / stderr 的子进程输出进入后台宿主日志。
- 轮转不 rename 打开的活跃日志文件；Windows 上活跃日志被持有时仍能完成轮转。
- truncate 轮转后，活跃日志从 0 连续增长，不产生稀疏空洞。
- 同一秒内连续触发多次轮转，不会覆盖已有轮转文件。
- `zz serve logs` 能读取当前活跃日志。
- `zz serve logs --tail` 在日志轮转后继续跟随新的活跃日志。
- 旧路径 `<ZHIXING_HOME>/server.log` 不会和新路径长期并存成两个有效日志源。
- `zz --log` 的诊断 dump 行为不被后台宿主日志治理混淆或破坏。
- 单元测试覆盖策略计算、轮转、保留清理、旧路径迁移、读取器跨轮转跟随。

## 执行计划

本改造应拆成多个独立可提交单元，不做一次性大提交。原因是：路径策略、生命周期算法、daemon 接入和 CLI 读取是不同层级的变化，混在一起会让问题定位和回滚都变差。

### 单元 1：日志路径与策略模型

目标：先建立后台宿主日志的稳定模型，不接入 daemon。

范围：

- 新增后台宿主日志路径模型：
  - 活跃日志：`<ZHIXING_HOME>/logs/server/server.log`
  - 轮转日志：`<ZHIXING_HOME>/logs/server/server-YYYYMMDD-HHMMSS-SSS-<seq>.log`
- 新增默认策略常量：
  - 活跃文件上限：8 MiB。
  - 轮转文件最多保留：5 个。
  - 轮转文件最长保留：14 天。
  - server 日志目录总量上限：64 MiB。
- 新增轮转文件命名函数，保证同一毫秒 / 同一秒多次轮转不覆盖旧文件。
- 明确旧路径 `<ZHIXING_HOME>/server.log` 到新路径的迁移 / 兼容规则。

不做：

- 不改 daemon 启动。
- 不改 `zz serve logs` 行为。
- 不做真实 copy-truncate。

验收：

- 默认路径全部从 `ZHIXING_HOME` 派生。
- 轮转文件名 collision-safe。
- 旧路径和新路径不会被设计成长期双事实源。
- 纯逻辑单测覆盖路径、策略默认值、命名唯一性、迁移决策。

### 单元 2：生命周期治理

目标：实现后台宿主日志生命周期管理能力，但先不接入真实 daemon。

范围：

- 实现 `ServerLogLifecycle`。
- 支持启动期清理。
- 支持活跃日志大小检查。
- 支持 copy-truncate 轮转：
  - 复制当前活跃日志到唯一轮转文件。
  - 对活跃日志执行 truncate。
  - 不 rename 打开的活跃日志文件。
- 支持保留清理：
  - 过期文件优先。
  - 超数量再删。
  - 超目录总量最后兜底。
- 明确清理 / 轮转失败语义：
  - 不能阻塞后台宿主启动。
  - 必须写入可观测错误。

不做：

- 不改变 daemon stdout / stderr 捕获方式。
- 不改 `zz serve logs` 用户入口。

验收：

- 活跃日志超过上限后会产生轮转文件。
- 轮转后活跃日志从 0 连续增长，不产生稀疏空洞。
- copy-truncate 不依赖 rename 打开的活跃文件。
- append fd 不变量有测试或明确断言保护。
- 保留数量、保留天数、目录总量均被测试锁住。
- 清理失败不抛穿主流程，但错误可观测。

### 单元 3：接入 daemon 与 `zz serve logs`

目标：把生命周期治理接到真实后台宿主，并保持读取体验稳定。

范围：

- daemon 父进程打开新的活跃日志路径。
- 保持 fd 级 stdout / stderr 捕获，不退化成只捕获显式 logger。
- daemon child 启动早期启动 `ServerLogLifecycle`。
- PID / status / RPC server 信息里的 logPath 指向新的活跃日志路径。
- `zz serve logs` 默认读取新的活跃日志路径。
- `zz serve logs --tail` 支持轮转后继续跟随活跃日志。
- 处理旧路径 `<ZHIXING_HOME>/server.log`：
  - 迁移或一次性兼容读取。
  - 不让旧路径和新路径长期并存成两个有效日志源。

不做：

- 不把 `zz --log` 和后台宿主日志合并。
- 不增加用户级日志配置参数。
- 不改后台宿主日志的产品定位。

验收：

- `zz serve logs` 能读取 daemon 的当前活跃日志。
- `zz serve logs --tail` 在 copy-truncate 后继续输出新增日志。
- daemon 的 `console.log`、`process.stderr.write`、Node warning、未捕获异常栈进入后台宿主日志。
- 继承 stdout / stderr 的子进程输出进入后台宿主日志。
- `zz status` / server status 展示的 logPath 与实际活跃日志一致。
- 旧路径不会继续作为新 daemon 的写入目标。
- `zz --log` 诊断 dump 行为不受影响。
- CLI 构建通过。

### 收尾验证

完成三个单元后执行一次收尾验证：

- 受影响单测全量。
- CLI 构建。
- 手动验证后台 daemon 启动后日志路径、查看、tail、轮转后的继续读取。
- 检查 `~/.zhixing/server.log` 不再作为新 daemon 的长期写入目标。
