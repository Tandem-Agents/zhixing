# 生命周期概念定义与规范 (Lifecycle Concepts)

> **定位**:把知行中散落在 run / turn / 段(注意力窗口) / 会话 / runtime / 子 agent 等处、
> 彼此重叠又容易混淆的「生命周期」概念,收敛成一套清晰、互不矛盾的定义与边界——
> 作为后续机制设计与术语使用的单一参照。
>
> **范围约束**:本文只做**概念定义与边界澄清**(每个生命周期是什么、从哪开始到哪结束、
> 与相邻概念怎么划界),不展开具体机制实现(那些留到对应模块级 spec)。

---

## 维护约定

- **逐条对齐**:一个概念一节,与用户逐个对齐后落定;未对齐的**不擅自填、不捏造**。
- **事实锚定**:每个概念的"代码落点"必须能指到一手源码(文件/符号),声明面与生效面一致。
- **沉淀去向**:定义稳定后并入对应模块级 spec(避免权威分散);若本文成长为跨模块的
  长期术语参照,再评估是否在 [drafts-roadmap](../drafts-roadmap.md) 登记或升级为 spec。

## 一、待定义的生命周期概念

> 逐个对齐填入。建议每个概念覆盖以下维度(模板可在对齐时调整):
>
> - **名称**:概念名 + 代码中的对应物
> - **一句话定义**:它是什么
> - **出生**:在哪个时机、由谁创建
> - **存活**:存活期间的不变量与状态归属
> - **结束**:在哪个时机、如何终止 / 落盘 / 回收
> - **边界**:与相邻生命周期概念如何划界(谁包含谁、谁正交于谁)
> - **代码落点**:一手源码引用

### 注意力窗口（attention window）

注意力窗口是当前喂给 LLM 的上下文视图——物理层（append-only 的对话持久化，唯一真相源）的派生、瞬态投影，按模型优质注意力尺寸而非物理上限维护，生命周期单位为“段”。它在进入交互模式、上下文编排启动时由持久层倒读重建出首个窗口，此后每次上下文重构完成即新生一个；存活期内 system prompt 与 tools、历史对话列表 字节恒定以保 cache。它终结于“对整个上下文的重构”——注意力触顶自动切段、或用户 `/compact` 主动触发（`/clear` 清空、`/resume` 换对话为同类变体），此刻 cache 前缀失效，故 system prompt 中的 skill 列表等也可搭此边界一并重建。

### 轮（turn）

一轮（turn）是一次 LLM 调用、连同该次调用所触发的工具执行——两者合为一轮，以 LLM 调用次数计数（每完成一次调用 `turnCount` 自增）。一个用户消息可引发多轮：LLM 每调用一次，若请求了工具便执行、并把结果回灌为下一次调用的输入，如此往复。一轮终结于该次调用所请求的工具执行全部完成之时（触发 `turn_complete` 事件）；若某次调用未请求任何工具，则此轮即末轮，随最终回复产出而结束。

### run

run 是一条用户消息的完整往返——一次 `runtime.run()` 调用，内含连续的多个 turn，边界为 `run_start`→`run_end`。它在 `runtime.run()` 被调用、用户消息作为入参送入时出生，驱动主循环一轮接一轮地跑（每轮的工具结果回灌为下一轮输入），直到某轮 LLM 不再调工具、产出最终回复。最终回复（或中断、出错、触达最大轮数）令主循环退出并返回 `RunResult`，run 随之结束，其间创建的 `eventBus` 等 per-run 资源一并回收。

## 二、从需求出发看生命周期

> 从实际需求倒看：在哪些生命周期边界上需要能介入做事。本节只记录需求本身，机制实现另行展开。

1. **注意力窗口开启**（对应用户的第一条消息）——需要能在第一条消息中注入内容，或做其他准备工作。

   结论: **第一个钩子没有问题** ——需求合理、可实现、不跟任何现有功能冲突。
2. **run 开启**（对应一条用户消息）——需要能在该用户消息中注入内容，或做其他工作。
   结论: 没有问题,比第一个还干净
3. **run 结束**（其多轮 LLM 调用全部干完）——可以发现并总结经验。
   结论：也没有问题
4. **注意力窗口结束**——暂未想到用途，先预留接入点。
   结论：也没有问题

### 要求

1. 现有实现中可能存在与这些需求语义不同、却类似甚至重复的功能，其中一些已经承接了部分需求。设计新方案时须逐一甄别：它们能否被新的生命周期方案统一收编；凡能收编的就一并处理干净、不留新旧并存——多套语义重叠的方案即是架构债务。
