# 创建技能：从"委派子 agent"到"能力内化"

> 设计思想记录。两个方案的对比，和为什么换。

## 一、最初的想法：委派给一个子 agent

把"创建技能"做成一个独立的子 agent。用户想存个技能时，主 agent 把这活外包出去：子 agent 拿着专属的提示和工具，自己起草、自己处理，干完把结果交回主 agent。用户不直接碰子 agent，中间由主 agent 转述——要改哪儿，用户跟主 agent 说，主 agent 再派一次活给子 agent。

一句话：主 agent 是包工头，子 agent 是隔间里的专家，用户隔着包工头喊话。

## 二、更妙的想法：把能力内化进主 agent

不造子 agent。把"创建技能"变成主 agent 自己就会的一件事，由两样东西承载：

- 一份**方法**：怎么把一件事提炼成好技能。不常驻，用到时才加载。
- 几个**工具**：把方法落地（起草、保存）。

用户从头到尾只跟一个助手对话。说一句"把这个存成技能"，助手当场就在对话里拟草稿、跟你来回改、存好、继续——没有召唤、没有切换、没有进出场。同一份能力，助手能自己在合适时机用，用户也能用一条指令直接唤起。

一句话：不是把活外包出去，而是让助手自己学会这件事、当面陪你做。

## 三、为什么排除最初的委派方案

创建技能本质是**创作和打磨**——反复看草稿、说哪不对、改了再看，要用户全程在场、紧凑来回。委派恰恰卡在这上面：

- 隔着主 agent 转述，你的意思会走样，像传话游戏。
- 每改一轮都多一层 agent 往返，又慢又费。
- 草稿在两个 agent 之间来回搬，把主对话也撑乱了。

根上是**用错了抽象**。委派适合"给个明确任务、要个结果、过程不用管"的活——调研、批量处理、独立审查，隔间里干完交回正合适。但创作打磨要的是并肩做，不是隔墙喊话。

判据：**要不要委派，看这件事用户需不需要全程参与。** 需要参与的，内化进助手、当面来回；只要结果的，才外包出去。
