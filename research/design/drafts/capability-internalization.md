# 创建技能：从"委派子 agent"到"能力内化"

> **性质**：需求确认文档（**已定稿**）——碎片经多轮裁决收敛，回答"要什么、为什么"；架构细节见末节「落地归属」指向的 spec。
> **上游**：[skill-module.md](./skill-module.md)（skill 需求定稿）、[skill-authoring.md](../specifications/skill-authoring.md)（创作 spec，随本文档定稿重写）。
> **取代**：[skill-new-ux-redesign.md](./skill-new-ux-redesign.md) 的 alt-screen 编辑屏方向——触发背景见 [screen-rendering/overview.md](../../internals/screen-rendering/overview.md)（alt-screen 无原生滚动、无状态层，长草稿"反复看、改了再看"撞硬天花板）。

## 一、本质与方向

**两个方案，一个判据。** 最初想把"创建技能"委派给子 agent（主 agent 当包工头、用户隔着转述喊话）；后来收敛为编辑屏（进出场的全屏 modal）；最终都被同一个判据否决：**要不要委派 / 切屏，看这件事用户需不需要全程参与。** 创建技能本质是创作和打磨——反复看草稿、说哪不对、改了再看——需要用户全程在场、紧凑来回。委派隔着传话走样，切屏强迫语境切换，都是用错抽象。

**方向：能力内化，且内化到底——功能点消失、机制下沉。** 用户面前没有"新建技能"这个功能（没有专门入口、没有屏）；系统里有一条把不变量焊死的工具管线。用户说一句"把这套存下来"，助手当场在对话里拟草稿、来回改、存好、继续。

**这是一套机制，不是专门的东西。** 现在做的不是模型自主生成 / 迭代技能（那是 v2），而是**通用的能力内化机制的首个承载**：未来会有更多能力以同一模式内化给模型——机制只有一个（索引 → 唤醒 → 按需加载方法 → 工具落地），能力只是内容。模型越强，能力越该以"知识"形态交付（教方法）而非"代码"形态（写死流程），而知识形态在有限上下文里的唯一可扩展解就是渐进披露——机制与行业演化同向，经得起时间检验。

## 二、需求梳理（碎片，已裁决）

### 机制定位（通用内化机制，技能创建是首个承载）

- **系统能力 = 内置技能，机制自举**：「怎么把事提炼成好技能」这份方法本身就是一份程序性知识，与用户技能同走渐进披露管线（索引常驻稳定前缀 → 命中唤醒 → 按需加载全文 → 工具落地）；机制能承载自己，证明抽象选对
- **上下文纪律是硬约束**：暴露给模型的只能是索引 / 唤醒钩子（一行级），能力细节按需加载，绝不常驻
- **新增能力零新机制**：未来内化更多能力 = 加一份方法 + 登记工具；来源标记天然区分（系统 / 官方来源，技能管家不碰——信任边界插座已预留）
- **系统能力与用户资产硬边界**：builtin 不进 `/skills` 管理列表（机制内部件，非用户资产）、**不注册 `/<name>` slash**（否则系统能力变相恢复成用户命令入口；其唤醒只有模型自主索引命中 + 用户自然语言两路）、不可删、技能管家不碰；索引预算**分池**——builtin 占固定小额度，不挤占用户技能 top-N；定制走既有 copy-on-write（fork 到 own 区即成用户资产，进列表、占用户预算）
- **工具 schema 机制级答案：注册表恒定、暴露形态可演化**：新增**能力工具注册表**（能力 → 工具 + schema + 不变量钩子的登记处，暴露形态无关）。v1 工具直接进 `tools[]`（红线：每能力 ≤ 1 工具，优先零工具）；能力增多（> 3 或工具字节超阈）切换为"常驻单一执行网关 + schema 随方法文档按需加载进对话流"，网关按注册表硬校验执行——`tools[]` 恒定不膨胀、窗口内 byte-equal 永不破、校验始终是硬护栏。预留的是注册表数据结构，不是网关本身（v1 不为一个工具套间接层）

### 形态与入口

- **不暴露专门创建功能点**：`/skill-new` 撤；用户自然语言表达意图即完成，无召唤、无切屏、无进出场
- **形态 = 方法 + 保存工具，起草无工具**：方法（不常驻、按需加载）教模型怎么提炼；起草是模型行为——草稿就是对话里的一段话，没有起草工具、没有 JSON 中间格式、不花单发 LLM 调用（那是编辑屏时代表单渲染的遗产）；判断全交模型、护栏全在保存工具
- **对话流即创建界面**：草稿以 markdown 渲染落对话流（scrollback 原生滚动），修改就是说话，保存是一次工具调用
- **创建与打磨是同一能力：save_skill 为 upsert 语义**——同名即更新（经同一"明确接受"护栏）；"把那个技能补一条坑"走对话、同一工具、同一不变量焊接点，v2 技能管家迭代亦走此管线。内容修改走对话（本能力），结构管理（删 / 禁用 / pin）走 `/skills`（不在范围）——互补不重叠
- **"使用"入口保留、"创建"入口不设**：`/<name>` 手动唤起照旧（使用 = 高频确定性动作）；创建 = 低频创作动作，对话合适——不对称是有意的
- **发现性靠主动提议，提议有边界**：仅两种时机——用户显式要求；或一套做法刚被验证走通且置信足够高。**只提议、绝不自动保存**，被拒不就同一件事重提（防打扰；自动产生是 v2 技能管家的事，带门槛、在后台）。边界写进内置方法本身
- **保存必须有用户明确接受，双层焊死**：产品层——方法明文规定保存前须拿到明确同意（"存吧 / 就这样"），意图不明就问一句；系统层——save_skill 是持久化副作用工具，按既有安全管线走（✎ 副作用锚 / 确认策略），不开绕行特例。静默落盘在机制上不可能
- **全接入面可用（验收项）**：能力装配在 runtime 层，cli / serve / 飞书 / ephemeral 经同一管线触发，不开 cli 专属通道

### 系统不变量（"直接不管"不行的原因）

- **四类不变量焊死在保存链路里**（通用 Write 给不出，专门屏也不比工具多任何保证）：① 凭证脱敏（secret 绝不固化进技能）② 来源落位（恒写本地区，目录即来源）③ 索引一致性（技能集变更触发稳定前缀索引重建）④ 落盘格式与分区（标准 SKILL.md / 模式标注）——这四类无触发语义、归 SkillSavePipeline；**归属语义（stewardCreated，v2"只动自己造的"信任边界）是触发方语义，由包装层分别处理**（save_skill 用户路径 / StewardWriter 管家路径），Pipeline 不感知触发方

### 管理与控制

- **`/skills` 管理器不在本次范围**：管理路径（manager-command → manager-screen）与创建路径零依赖，本次不动；"管理走对话 + 工具、长内容阅读编辑还给用户编辑器"是**远期方向**（cli 是交互平台、阅读编辑非长项），待内化机制验证后另行裁决
- skill-module 中"控制面复用 resume / mcp 既有交互流程"的表述随本方向修订

### 承接与退役（边界：仅创建路径，skill 模块内）

- **创建路径退役**：`/skill-new` 入口、editor-screen、editor-controller 阶段机；manager-screen（`/skills`）与 admission（`/skill-add` 外部接入）均不在列——三条路径相互独立
- **其他模块零波及**：config-editor 等 alt-screen 屏独立自管、零共享；`composeViewport` 是 tui 公共原语，保留
- **UI 无关内核原样保留**：scrubSecrets 脱敏、admission 审查、SkillStore 落盘——换载体不换内核；旧起草 prompt 不保留形态、其提炼判据融入内置方法（与"起草无工具"一致）
- **UX 评审产品智慧平移进对话流**（与载体无关）：起草前"收自:X"语境知情、脱敏可见（真抹了才说）、诚实承诺（只说 v1 真有的：`/<name>` 唤起 + description 检索命中）、放弃 / 覆盖前护栏确认
