# 待做需求模块灵感池

> **性质**：活文档，只记录尚未落地的需求模块、灵感、思路和待收敛信息。
> **范围约束**：这里只放“还没做、还值得保留”的东西；已经完成、废弃、迁入正式 draft / spec / roadmap 的内容应从本文清理。
> **维护原则**：宁可短、准、可继续追问；不要把本文写成长期规格、实现计划或历史归档。

## 维护规则

1. **只记录待做**：做完、放弃或迁入正式文档后，删除对应条目。
2. **先保留灵感原貌**：早期想法可以粗糙，但要保留触发点、用户价值和未决问题。
3. **不急着架构化**：没有收敛前不强行拆包、定接口、排期或写实现方案。
4. **及时迁出**：某个模块进入认真设计时，新建独立 draft，并在本文移除原条目。
5. **避免历史堆积**：本文不是决策记录；需要保留演进过程时，迁到对应需求确认或架构草稿里。
6. **分隔独立需求**：每个独立需求模块之间用 `---` 分隔，便于后续追加、迁出和整体删除。

## 记录格式

每个条目建议只写到足够恢复上下文，不追求完整：

```text
### <需求模块名>

- **触发**：
- **用户价值**：
- **灵感 / 思路**：
- **未决问题**：
- **下一步**：
```

## 待做需求模块

### 多视角发散收敛

- **本质需求**：复杂、关键、易误判问题需要一种内功能力：临时拉起多个只读继承主注意力窗口的隔离子 agent，从不同视角独立产出、交叉吸收，再由独立收敛 agent 给出一份可执行判断。中间过程瞬时、不持久化、不污染主线；先做可手动/可被 AI 调用的基础设施，数量有硬上限，角色/组合后置。
- **运行骨架**：触发时先取主注意力窗口的只读快照；分配节点按当前问题动态切出 N 个视角（默认少量、硬上限）；各子 agent 基于同一快照独立产出，交叉优化一次后，由独立收敛 agent 生成唯一结论回主线。
- **`@` 交互**：`@` 是“召唤参与者”的统一提及面。`@ ` 后接问题表示召唤多个视角并发思考；`@名字` 表示召唤一个具体 agent / 角色（并可继续承载文件、对象等具名引用）。行头 `@` 直接生效；句中只有 `@` 前是空格才算触发，避免误伤 `email@x.com`。`@` 后无正文不触发；并发数量写在自然语言问题里，不用 `@5` 语法。第一阶段只做默认多视角，角色、组合、单角色临时唤醒后置。
- **触发定位**：手动 `@` 只是第一阶段发起者，未来 AI 自动触发应复用同一机制，未来是多种发起方式共存，注意架构设计。
- **与 `@文件` 并存**：`@ ` 召唤多视角，`@文件/路径` 仍按现有引用文件语义工作，二者共用 `@` 心智但由空格与具名目标区分。
