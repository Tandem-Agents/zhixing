# 多视角发散收敛能力需求确认

第一步：根据 “需求思考工作台”对能力a的理解，先思考要提供给用户怎样的使用体验；

能力 A 做好以后，用户不需要学习“工作流”、不需要手动拉多个 agent、也不需要自己设计并行流程。

用户提出一个复杂、重要、容易误判或需要高质量判断的问题时，知行会自然判断：这一步不能只靠单一路径直觉完成，需要临时拉开多个独立视角，再把结果收敛成一个可执行结论。

用户感受到的不是“多了一个功能”，而是：

1. 复杂问题不再只给一个单薄答案，而是明显经过了多角度思考。
2. 关键设计、审查、研究、决策节点更稳，少掉单模型视角盲区。
3. 不需要用户反复说“再从另一个角度想想”“找别人审一下”“并行比较几个方案”。
4. 输出不是多个答案的堆叠，而是一个收敛后的判断：推荐什么、为什么、放弃了什么、风险在哪里、下一步怎么做。
5. 只有当方向选择、风险承担或偏好取舍必须由用户决定时，才把问题交还给用户。

agent 的工作方式会变成：

1. 主 agent 在关键节点识别是否需要多视角发散。
2. 临时派出多个上下文隔离的子 agent，各自带着不同视角、约束或判断标准独立工作。
3. 子 agent 返回结构化结果，而不是随意聊天式答案。
4. 主 agent 比较分歧、识别互相冲突的判断、过滤低质量意见。
5. 主 agent 形成一个收敛结论，并把它接回当前任务主线。

这个能力的价值不是“并发”本身，而是让知行在关键节点自动获得更强的判断质量。

它应该是一个内功原语，不是用户入口；是一次任务中的临时增强，不是接管整段会话的流程系统；固定的是“发散再收敛”的工作形态，动态的是视角数量、视角内容、提示词、证据要求和收敛标准。

## 两个问题的结论(核实一手源码后)

一手事实:`runChildAgent` 现在唯一的生产消费者就是 Task 工具(workflow 已 revert);`loop-runner` 接收任意初始 messages、`AttentionWindow.getMessages()` 能导出主窗口当前快照、`buildSystemPrompt` 的 profile 是参数——但 `factory.ts` 把这三样写死成了「空白上下文 + 一句 "Begin." + 固定 `subAgentProfile`」。

**问题 1——为 subagent 做的地基要不要先做稳?不需要单独先做一轮加固,执行层已稳。**

结合 A/B 反推:A 要的并行派发、独立收敛、瞬时不留存、结果单向回,B 要的中断、事件可观测、安全边界、预算控制——全部落在现有执行层(loop-runner / EventBus lineage / 中断级联 / 子 broker + securityPipeline / ChildAgentResult 三态 / budget / ALS 隔离),A/B 直接复用,不用推倒、不用先补。真正的缺口不在"地基稳不稳",在 **dispatch 入口被 Task 写死**:① 继承主窗口(`getMessages()` + loop-runner 收任意 messages 已具备,缺 factory 别写死 "Begin.");② 可配角色 profile(`buildSystemPrompt` 收 profile,缺 factory 别写死 `subAgentProfile`)。把 dispatch 入口通用化(可选上下文策略 + 可配 profile)本身就是"做能力 A / 子 agent 模块"的核心工作,不是独立的前置阶段。设计时唯一要确认的集成点:dispatch 时刻能否拿到主 agent 当前窗口 owner 去取快照——能力在,可达性要接通。

**问题 2——现有 subagent 能力(Task)保留还是砍?现在不砍,底座更不能砍。**

分两层:**砍 Task ≠ 砍底座**。底座(runChildAgent / loop-runner / 事件 / 中断 / 安全 / 预算)正是 A 要复用的核心资产;workflow revert 之后,Task 是这层底座**唯一的现役消费者与验证场景**,砍了底座就成无人调用的孤儿。Task 这个入口本身也建议留,因为它有 A 替代不了的独特价值——**降噪隔离**:子 agent 不带主记忆、把噪音大 / 输出大的脏活关在子里、只带回干净结论(读超大文件提要、海量搜索结果筛选);而 A 恰恰相反,是带着主上下文做多视角。两者互补:**Task = 输入降噪隔离,A = 上下文共享多视角**,知行两个都需要。触发率低是"引导 / 何时该用"的问题,不是存废问题。最优路径:做 A 时把 dispatch 通用化,Task 降级为「空白隔离」策略之一、A 是「继承窗口多视角」另一种策略,共享同一底座;Task 入口的最终去留,交给 A/B 落地后的真实使用数据决定,不现在凭感觉砍(会丢底座唯一验证 + 丢降噪能力 + 无数据支撑)。

## 能力 A 需求确认(收敛正式版)

> 取代开头初版「做好后的效果」(那版默认了 AI 自动判断,已被推翻);中间「我的想法 / 已收敛 / 两个问题 + 结论」为决策演进记录,本节是收敛后的正式需求。

**用户价值(做好后的效果)**:用户不需要学习"工作流"、不手动拉 agent、不自己设计流程。在复杂、重要、容易误判、需要高质量判断的问题上,知行临时拉开多个独立视角、再收敛成一个可执行判断;用户感受到的是判断质量更强,而非"多了一个功能":

1. 复杂问题不再只给单薄答案,明显经过多角度思考。
2. 关键设计 / 审查 / 研究 / 决策节点更稳,少掉单视角盲区。
3. 不必反复说"再换个角度想想""找别人审一下"。
4. 输出不是多个答案的堆叠,而是一个收敛判断:推荐什么、为什么、放弃了什么、风险在哪、下一步怎么做。
5. 只有方向、风险承担或偏好取舍必须由用户决定时,才把问题交还用户。

**第一阶段形态(关键:不赌 AI 自动判断)**:

- 先做成**中立基础设施**:并行执行多个上下文隔离子 agent + 一个独立收敛 agent。可由 AI 调用、也可由用户手动触发;**具体由谁触发是后续产品选择,第一阶段不做 AI 自主判断**(AI 自主判断 = 不确定 + 不透明 + 用户莫名多倍消耗;而用户自己触发,这个行为本身就是意图)。
- 子 agent **继承主 agent 注意力窗口的只读快照**作为初始上下文(要懂上下文才能对"这个问题"给出有意义的视角),在隔离上下文里发散,瞬时、不持久化、绝不写回主窗口;**只有最终收敛结果进主 agent**。
- **收敛由一个独立 agent 完成**,不把 N 份子结果灌给主 agent(主 agent 上下文珍贵)——主 agent 只收一份收敛判断。
- **成本闸门**:每个子 agent 带一份主窗口快照 = N 份 token,所以并发数量、组合、内置默认上限与死的最大边界是必须的控成本设计。

**独立子 agent 模块(知行目前没有)**:内置一批有鲜明特征、真正带来不同思维框架的角色(乔布斯 = 产品直觉与极致取舍、爱因斯坦 = 第一性原理、女娲 = 创造性重构——名人只是载体,价值是那个真不一样的视角框架,不是模仿秀);用户可自定义角色、可搭配组合、可控并发数量。注意:"用户能自定义角色 / 搭组合"与"何时触发这个能力"是两件正交的事,不要混为一谈。

**复用与边界**:复用现有 subagent 执行底座(loop-runner / 事件 / 中断 / 安全 / 预算),只把被 Task 写死的 dispatch 入口通用化;Task(空白隔离子任务)作为另一种 dispatch 策略并存,不砍。

**定位**:它是内功能力,不是用户要学的流程系统;固定的是"发散 → 收敛"的工作形态,动态的是视角数量、视角内容、提示词、证据要求和收敛标准。
