# 技能（Skill）模块 — 需求确认

> **性质**：需求确认文档（需求已定稿）。**架构实现规格**见 [skill-system.md](../specifications/skill-system.md)。
> **事实依据**：[skill-system-comparison.md](../../source-analysis/skill-system-comparison.md)、[openclaw/skill-system.md](../../source-analysis/openclaw/skill-system.md)、[hermes-agent/skill-system.md](../../source-analysis/hermes-agent/skill-system.md)
> **不在范围**：① **与记忆模块的边界对齐 / 耦合**——skill 独立设计、照最好做，记忆未来可能重构、现在迁就即债务；② **多来源作用域分层 / 同名覆盖优先级**——OpenClaw 面向多项目分发的机制，个人助手无此诉求；可见范围已由「模式分区」吸收，最小覆盖已由「写隔离 · 本地同名优先」满足。

## 1. 本质与核心

**本质 —— skill 是第三种东西。** 工具是"能做什么"（read/write/bash 等原子能力），记忆是"关于用户与世界的事实"，skill 是**可复用的"做某类事的方法"（程序性知识），按需调取**。它回答的不是"能不能做"，而是"怎么把这类事做对"。真实需求在此：通用模型不缺能力，缺的是"你这摊事的特定做法"——你的约定、那套多步流程、踩过的坑；每次重讲既费 token 又不稳定，skill 把"做对的方法"沉淀一次、不必再教即可一致复用。

**核心机制 = 渐进披露。** 上下文塞不下所有方法，所以永远只在稳定前缀挂一个**廉价索引**（有哪些技能），命中了再**按需加载全文**（怎么做）——"无限方法库 + 有限上下文"由此共存。这是 OpenClaw / Hermes / Claude 三家独立收敛到的同一答案。

**核心价值 = 一致 → 累积。** 对个人助手，差异化不在"会很多技能"，而在**越用越懂我**：把"我的做事方式 + 过去的教训"累积成可复用资产，在恰当时机即时注入。静态库只到"一致复用"；能自建 / 迭代 / 淘汰的活体库才到"累积成长"。

**站在未来看。** 模型越聪明，"手把手写清每一步"的 skill 越贬值（聪明模型自己会）；durable 的价值收敛到两点：(a) 模型无从知道的**你的特定事实 / 约定 / 上下文**，(b) 真金白银换来的**别再踩的坑 / 最优路径**。趋势是 skill 从"详尽 how-to"**瘦成"指向特定性 + 沉淀教训"**，并越来越**由 agent 自己写下、自我进化**。

**与知行原则一致。** skill 不是因"模型不可信"才把每步焊死（那是脆弱、膨胀的反模式），而是给**聪明的模型**在**恰当时机**补上它无从得知的**事实**，判断仍交给模型——skill = 接地于真实特定性的、即时的程序性上下文。

## 2. 需求梳理（碎片）

> 本节只罗列需求碎片与方向，**不做拼接 / 取舍 / 去重**——冲突的碎片先并列摆着，留给后续章节裁决。三个来源：借鉴两家 · 第一性原理 · 用户视角。

### 2.1 借鉴两家（各取所长）

**从 OpenClaw 拿：**

- 索引格式与 Claude Code Agent Skills 对齐 → 生态兼容，能直接吃社区 / 官方技能
- 多来源层叠 + 按 name 去重的优先级（bundled < managed < 全局 < 工作区）→ 全局 / 项目分层覆盖
- 安装期静态扫描 + workspace 逃逸防护 —— 外来技能的第一道闸
- 索引快照剥离全文 + 字符预算降级 —— 上下文预算治理

**从 Hermes 拿：**

- **自主进化三层**：回合后台复盘 fork 新建 / 迭代 · 使用遥测状态机（active/stale/archived/pinned）· 周期 curator 合并 / 泛化 / 淘汰 —— "越用越懂我"的引擎
- **write-origin provenance 信任边界**：自动化只动"自己造的"子集，人 / 官方来源 off-limits —— 自我进化敢放手的前提
- skill 全文走**专用按需加载工具**（可带模板 / 内联渲染），而非塞进通用 read
- 两层缓存（进程内 LRU + 磁盘快照 mtime/size 校验）+ 复用父 system prompt 命中前缀缓存
- 本地可写 + 外部只读（`external_dirs`），自建只落本地

### 2.2 第一性原理（我们自己推导）

- 索引必进**稳定前缀**、且只在技能集变更时重建 —— 缓存友好是硬约束、非优化
- 渐进披露不可绕过 —— 无限方法库 vs 有限上下文的唯一解
- 触发靠**模型自选**（读索引后判断），不靠硬编码 substring —— 旧实现的教训
- skill ↔ memory 边界要划清：skill = 怎么做（程序性）、memory = 是什么 / 是谁（事实性）
- 自我进化是个人助手的**核心差异化**而非锦上添花；但必须建在信任边界之上
- 阻断策略不能只拦 critical（OpenClaw 的 warn 放行是留白）；引入分发需要**真验签**（两家都留白）—— 我们可超越处
- skill 要能被多步流程 / 多 agent 复用 —— 解耦边界要预留（即便初期单一使用者）
- **凭证零接触**：skill 全文 / 自进化产物绝不内联 secret —— 承接身份层不变量

### 2.3 用户视角（用户要什么）

- 别再每次重教"我这摊事怎么做" —— 一次沉淀、长期一致复用
- **越用越懂我**：我的约定 / 流程 / 踩过的坑被记住，并在对的时机自动补上
- 两种沉淀都要：我**手动写**一个（显式），也让 AI **自动帮我攒**（无感）
- 看得见、管得了：有哪些技能、能编辑 / 删 / 临时禁用 / pin 住
- 别乱触发乱注入 —— 可控、可预期、可信
- 想用别人 / 社区的技能，但要安全（来源可信、装前可审）
- 分层：全局通用 vs 单项目专属，互不污染

## 3. 知行的需求碎片（选定）

> 从 §2 逐条确认后选入。这些是**定稿的需求**；如何实现见 [skill-system.md](../specifications/skill-system.md)。

- **渐进披露（地基）**：索引（name + description + location）常驻系统提示词、不放全文；模型扫索引、命中后才用专用加载工具加载那一个的全文——先给目录、按需展开
- **格式统一**：采用公认标准 `SKILL.md` + YAML frontmatter（与 Agent Skills 生态一致）
- **外部接入**：提供外部 skill 接入能力，可加载符合该标准的社区 / 官方技能
- **外来技能审查**：装入前识别危险模式（规则扫描 prompt 注入 / 信息外泄 / 窃凭证 **+ AI 语义识别**）→ 结果**分级**：确凿恶意直接挡死、模糊可疑**交用户决定**，不靠系统阈值静默放行（超越 OpenClaw「只拦 critical、warn 放行」的留白）
- **目录边界约束**（防 symlink 逃逸）：访问技能目录内文件前，校验其 realpath 仍落在该技能目录内、越界即拒——防恶意技能用软链接读越界的系统敏感文件（如 credentials）
- **主动 top-N 索引**：索引**事前限定最大条数 N + 每条尺寸上限** → 总量有确定上界、**恒在预算内**（不存在"超出"，故**无降级兜底**；真超了是 N / 单条上限没设对的 bug，修 bug、不打补丁）
  - 取舍：超 N 时 **pin 强制进 + 其余按频次 / 新近度排序**取够 N；top-N 之外模型不自动见、**用户仍可手动唤醒**
- **自我进化（谨慎版）**：活体技能库，由**技能管家（Skill Steward）**自主负责产生 / 迭代 / 治理，全程建在来源边界上
  - 产生：低频触发 + 廉价门槛把关，过门槛才用 **main 档**复盘新建 / 迭代技能（高副作用动作，不交 light）
  - 治理：技能管家周期性用 **main 档**做合并 / 泛化 / 淘汰，带工作量预算（不无脑扫全库）
  - 淘汰防误删：删不删交技能管家 **判断**（内容 + 使用信号 + 是否被取代），**不设"X 天没用就删"死阈值**；删 = **归档可逆**，不物理删；低频仍命中 / pin / 用户写 / 官方 一律不动
  - 度量：极简使用信号（最后命中时间 / 次数），喂**两个消费者**——技能管家的**淘汰判断** + 主动 top-N 的**排序**；不暴露用户、不建状态机
  - 来源边界：技能管家 **只处理它自己（在自动复盘 / 轮次判定中）创建的技能**；其余一切——人手写的、人让 AI 在前台写的、外部 / 官方接入的——都算**用户的**，一律不碰。每个技能据此打来源标记（本质是「技能管家自主造的」vs「其它一切」的二分）
  - 凭证脱敏：产生 / 迭代技能时必须过滤掉对话里冒出的 secret，**绝不固化进技能**——技能会反复加载进上下文、且设计上可分享，secret 混入的危害被放大；复用系统级脱敏、不自造（导出 / 分享时同理）
- **专用加载工具**：命中后用专门的「加载技能」工具读全文（非通用 read）——既是使用度量的干净采集点，也支持模板 / 变量渲染
- **写隔离（来源边界的能力兜底）**：接入区**对技能管家不可写**——技能管家的写能力被限定在可写区、够不到接入区，从能力层兜住"只动自己产生的"（来源判定即便出 bug 也写不进）。注意这是**对技能管家的约束，不是文件夹权限锁死**：用户随时能改任何技能，改接入技能时自动 fork 一份到可写区改（copy-on-write），原件不动、可继续同步上游，同名优先让改后版本生效（fork 后即算"用户的"，技能管家更不碰）
- **模式分区（索引分发，正交于存储与可写性）**：给每个技能标它属于哪个模式 **main / work**（按技能标注、非文件夹），决定其索引注入哪个模式的系统提示词——**标 main 的进 main 模式、标 work 的进工作场景模式**；每个模式只见相关索引、模型更专注（与「主动 top-N」配合：模式分区先按场景过滤、top-N 再在模式内按价值取舍）
  - 归属确定、不靠模型判断：**技能管家**产生的按**产生时所处模式**自动定（main 模式产生→main、工作场景产生→work）；**用户**写 / 引入 / 让 AI 出的，由用户自己定 main 还是 work
  - 只管**自动注入**：用户在任何场景都能手动唤醒任何技能（系统提示词索引只是"按模式 + 分区过滤的视图"，全集始终可指名调用）
  - 现阶段 work **全进所有工作场景**；架构**预留**"未来 work 细化绑定到具体工作场景"，现阶段不做
- **可复用边界**：skill 子系统做成独立单元（索引 / 加载 / 审查 / 进化），主对话 / 工具循环 / 子 agent / 技能管家等都能调；现阶段只**预留解耦边界、不写死**到单一入口，不提前接多方
- **看得见、管得了**：用户对技能库的控制面——列出技能、编辑 / 删 / 临时禁用 / pin（钉住不被技能管家淘汰）；是「自我进化」的用户控制另一端。交互**复用 / 扩展现有 resume、mcp 那套既有流程**、不从零造
  - 手动唤醒范式：复用既有 slash 指令体系，每个技能可注册为 `/<name>`（可带参）

## 4. 落地分期（路线）

> 详细架构见 [skill-system.md](../specifications/skill-system.md)(v1 地基)与 [skill-evolution.md](../specifications/skill-evolution.md)(v2 自进化)。

- **第一版 · 地基**：静态 + 手动 + 模式分区的技能库（渐进披露 · 格式统一 · 外部接入 · 外来技能审查 · 目录边界约束 · 主动 top-N · 专用加载工具 · 写隔离 · 模式分区 · 可复用边界 · 看得见管得了）。
- **第二版 · 自进化**：技能管家（产生 · 治理 · 淘汰防误删 · 来源边界 · 凭证脱敏）。
- **铁律**：第一版必须预留好第二版要插的所有"插座"（度量信号 · 来源标记 · 专用加载工具 · 可重建插座），让第二版是纯增量、不返工。
