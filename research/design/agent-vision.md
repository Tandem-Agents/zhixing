# Agent 愿景草稿 (Agent Vision)

## 维护原则

**只关注"agent 作为知行核心概念"的愿景、产品理念与架构方向**,不维护变更记录、状态进度、历史演化等无效信息。

- **范围**:本文讨论的是 **agent**(智能体本身作为知行的核心抽象与产品形态),**不是** sub-agent(子 agent 是 agent 概念之下的一种派生/组合形态,属下游话题)。本文先把上游的"我们要的 agent 是什么"想清楚,再谈任何派生设计。
- **写什么**:调研(业界与第一性原理)/ 产品理念(知行的 agent 要承载什么、不承载什么)/ 架构方向(为支撑该理念,抽象与边界应如何分层)/ 待对齐的开放问题。
- **不写**:修订记录 / 决策时间戳 / 状态进度标签 / 对历史讨论的回应 / 实现细节(schema、字段、文件路径、API 形态——那些留到 spec 阶段)。
- **更新方式**:原地改,不追加历史段;结论变了直接覆盖,不留旧版本注释。历史与演化留给 `git log`。
- **协作方式**:分批渐进对齐,一次只推进一个 Phase——助理列该 Phase 的待定问题并附倾向,用户回复后助理把共识提炼为"对齐结果"(聚焦方向、不展开实现),再进下一 Phase。

## 调研

> 维度锁定:产品定位 → 主/子 agent 模型强弱分配 → 集群/组合形态。下列为各产品**设计方向缩略**(非官话),不预设知行结论。

**Claude Code(编程 agent)**:定位生产级编码、追求产出质量,故主循环用最强模型(Opus)做推理/架构/综合,子 agent 按任务降配——实现交 Sonnet,检索/只读探索默认 Haiku(便宜约 15×)。核心理念:把"必须强模型"的认知任务与"弱模型够用"的机械任务分离,强脑在上、廉价手在下;二维控制面 = 模型档 × 思考力。

**OpenClaw(通用个人助手)**:定位广覆盖、成本敏感,主张"没有最佳模型",主 agent 用中档六边形模型(Sonnet 级,接近 Opus 能力但成本几分之一)。模型无关、可路由;典型反向配置 = 强 orchestrator + 廉价子 agent(DeepSeek)省 40–60% 成本;设能力下限(弱模型会选错工具/打转)。理念:成本/能力/速度三角权衡,交由用户调。

**Hermes(自托管个人助手,Nous)**:定位隐私、无供应商锁定、长期陪伴复利成长。模型完全无关,一句命令换模型;子 agent 可按配置自动路由到不同模型,会话/终端隔离;极端隐私可本地 Ollama 跑、API 成本归零。理念:用户拥有模型选择与数据主权,agent 越用越强,反 SaaS 锁定。

**Kimi(Agent Swarm / agent 集群,Moonshot)**:核心是自组织集群——经强化学习(PARL)训练的 orchestrator 把目标拆成可并行子任务,动态实例化最多 100–300 个领域子 agent(各带定制 prompt+工具),三层 = 编排/领域 agent/聚合。理念:靠"训练出来的编排智能"而非 prompt,以动态分工 + 大规模并行换速度与规模;子 agent 是临时冻结实例。

**Manus(通用自主 agent)**:定位全自主完成项目,非对话/流程器。形态 = "高管带专业团队":orchestrator 把复杂任务委派给规划/检索/codegen/校验等专职子 agent 并行协作。模型 = 强核心(Claude)+ 微调廉价模型(Qwen)分担子角色。理念:不堆单一巨模型,以专业化团队换更细腻高效。

> 来源:Kimi Agent Swarm(kimi.com/blog/agent-swarm、datacamp)、Claude Code model-config(code.claude.com/docs)、OpenClaw 模型选择(getopenclaw.ai、deepinfra)、Hermes(hermes-agent.nousresearch.com、turingpost)、Manus 架构(workos.com、manus.im/blog)。
