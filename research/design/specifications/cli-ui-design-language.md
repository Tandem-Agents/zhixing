# 知行 CLI 视觉设计语言

> **状态**：📐 方案设计
> **对齐过程**：[problems/cli-ui-visual-foundation.md](../problems/cli-ui-visual-foundation.md)
> **适用范围**：所有终端 UI 面板（配置编辑器、REPL、未来 TUI 模块）

---

## 一、设计前提

知行是 personal AI agent，CLI 是日常陪伴的"家"——不是临时使用的脚本工具。视觉风格不能像 dev tool 那样冷峻，也不能像 chatbot 那样卡通——要有"工具的克制"加"陪伴的温度"。

参照产品：Claude Code（强 chrome）、opencode（极简 accent）。**两者都不照搬**——知行有自己的产品调性，需要自创设计语言。

## 二、用户场景与情绪锚点

设计语言反向推导自用户在每个场景下的核心情绪需求：

| 场景      | 用户情绪需求 | 设计含义                                            |
| --------- | ------------ | --------------------------------------------------- |
| 首次配置  | **不慌**     | 步骤明确、可后悔、清楚下一步是什么；视觉重量轻      |
| 日常对话  | **专注**     | 视觉不抢戏、不打扰思路；只在必要时提示              |
| 长会话    | **安心**     | 上下文余量可见、状态透明、不会突然爆                |
| 错误/中断 | **可控**     | 错在哪、能怎么办、不害怕；红色不刺眼                |

## 三、七条核心原则

### P1 · 安静而非热闹

- **低饱和**：颜色不刺激、不闪烁、动画克制（spinner 80ms 已是上限，不再加更多动效）
- **少即是多**：默认状态信息少、需要时才出现（如上下文 < 50% 不显示百分比）
- **拒绝表情包/卡通**：知行是工具不是玩具——不引入 Claude Code 风的角色 emoji 装饰

### P2 · chrome 用于"边界"，留白用于"层级"

- box drawing 圆角框只在**容器边界**用：Welcome/Header、Input 编辑面板这种"独立空间"
- 章节、入口、按钮的层级靠**留白 + 字符前缀**（`◆` 章节 / `▸` 选中 / 缩进）表达，不堆 chrome
- 避免 box 套 box 嵌套——chrome 多了反而压缩信息密度

### P3 · 状态可识别不依赖颜色

- 状态用 **icon + 文本** 双通道：`⚠ 待补 API Key` / `✓ 已配齐` / `· 未启用`
- 色弱用户、终端配色不同时仍能识别
- 颜色是**强化**而非**唯一**信息载体——通过率优先于美观

### P4 · CJK 优先且永远对齐

- 中文是第一语言（"知行"两字本身有书卷气，UI 不应英文味重）
- 英文混排时路径/文件名/命令用 dim 灰色降权，让中文主导视线
- 列宽严格按 CJK 全宽 = 2 列计算——绝不出现右边参差
- 实现引用：`packages/cli/src/tui/line-width.ts`

### P5 · 单一品牌主色（青绿系）

- **一个**主色作为 brand accent，统一用于：选中态、品牌标识、主操作按钮
- 选**青绿系**（cyan-teal 区间）：稳重、非娱乐、契合"知行"调性；与 yellow（待办）/ red（错误）/ green（成功）拉开层级
- 收敛现状散落的 `cyan / blue / cyan.bold` 多色滥用——所有"被关注但非状态"的元素归一到此色

### P6 · 按钮形状化，操作层级清晰

- 按钮用 box drawing 边框包裹（`┌──┐ │ │ └──┘`），与文字行天然区分
- **主按钮**（如"完成"）：绿色边框 + 当前状态绿色文字（在"全部就绪"时形成视觉路径）
- **次按钮**（如"取消"）：dim 边框，视觉重量轻
- 选中态：反白 + 粗体（不再仅靠 `▸` 一个字符）

### P7 · 输入区是独立视觉容器

- 输入行**不是**普通文本行——用形状让用户一眼识别"这里可输入"
- 走自适应宽度框（与 P2 "chrome 用于边界"一致：输入区就是"独立空间"）
- 框宽度跟随终端列数，绝不写死；框内**输入光标位置稳定**（不被状态文本挤动）
- **超长输入框内自动换行**——绝不横向滚动；框高度按内容按需扩展，保留全部输入可见
- 适用面：config-editor 的 input/add-model 面板，REPL 的 prompt 行
- 具体形态规范（chrome 字符 / 多行扩展 / 提交回显降级 / typeahead 选中行点阵纹理同源）见 [input-zone-visual.md](./input-zone-visual.md)

## 四、视觉元素规范

### 4.1 主色（brand accent）

- 青绿系（cyan-teal）
- 用途：选中标记 `▸`、品牌标识 `✦`、主按钮边框、强调文字
- 不用于：状态文本（状态走 yellow / red / green / dim）

### 4.2 chrome 容器

- 形态：圆角框 `╭─ Title ──...───╮ ... ╰─...───╯`
- 标题嵌入顶边：`╭─ 知行 · 首次配置 ─...───╮`
- 宽度：自适应终端列数，左右各留 1 列余量
- 适用：Welcome/Header 区、Input 编辑面板、Add-model 面板

### 4.3 状态 pill

- 格式：`<icon> <文本>`，icon 与文本之间一空格
- 三态映射：
  - 已配齐：`✓ <text>`（绿）
  - 待补 / 阻塞：`⚠ <text>`（黄）
  - 未启用 / 不需要：`· <text>`（dim）

### 4.4 按钮

- 形态：`┌────...────┐` / `│ <label> │` / `└────...────┘`
- 主按钮：边框绿色，label 跟随状态着色
- 次按钮：边框 dim
- 选中：label 反白 + 粗体
- 宽度：根据 label 长度自适应，最小宽度保证视觉重量

### 4.5 章节

- 头：`◆ <title>`（粗体 + 主色）
- 描述：紧贴头下方 dim 行
- 入口行：缩进 2 字符 + `▸` 选中标记
- 章节间：1 空行；入口间：0

### 4.6 全宽分隔

- Footer 之前用全宽 `─` 分隔
- 宽度：终端列数

### 4.7 提示文字

- Footer 提示行：`↑↓ 选择 · Enter 进入/确认 · Ctrl+C 退出`，dim
- 帮助提示：必要时才出现，不堆叠
