# Token 用量显示策略

> **状态**: 📐 方案设计（2026-04-09）
> **前置**: Agent Loop、Context Engine、Session 持久化已完成
> **关联**: context-engine.md（预算事件）、session-persistence.md（会话追踪）、ADR-005（CLI 架构）

## 零、设计动机

当前实现在每轮结束后固定显示 `─ 入 1,614 · 出 79 tokens · 1.9s`。

**问题：**
1. 原始 token 数字对用户无意义 — 1,614 是多还是少？不知道
2. 不含参照系 — 没有总容量，无法感知"剩余空间"
3. 永远显示 — 即使第一句"你好"也输出 token 统计，是噪音
4. 包含 system prompt — 但用户无法改变它，显示它没有行动价值
5. "入/出"标签虽然比箭头好，但仍然是面向开发者的术语

**用户真正关心的三件事（按优先级）：**
1. **系统在响应吗？** → 耗时反馈
2. **我的上下文快用完了吗？** → 容量感知（可行动的信息）
3. **这花了我多少钱？** → 成本意识（仅 API Key 用户关心）

## 一、竞品方案对比

| 维度 | OpenClaw | Claude Code | Cursor |
|------|----------|-------------|--------|
| 每轮 token 行 | 默认关闭，需 `/usage` 开启 | 不支持（社区请求被拒 #29600） | 不显示 |
| 格式 | `Usage: 1.6k in / 73 out · est $0.003` | 无 | 无 |
| 上下文容量 | TUI 底栏 `tokens 12k/200k 6%` | `/context` 彩色网格 | 不明确 |
| 费用 | `est $0.003`（仅 API Key） | `/cost` 命令 | Dashboard 网页 |
| system/user 区分 | 不区分 | 不区分 | 不区分 |
| 条件显示 | 无（有即显示） | `used_percentage` 首次 API 调用前为 null | 上下文多了才显示 |

**行业共识：**
- 三大产品都不区分 system token 和 user token
- Claude Code 直接拒绝了每轮显示 in/out 的功能请求
- 上下文容量（%）比原始 token 数更有用户价值

## 二、设计原则

### P1：信息要可行动（Actionable Information）

"入 1,614 tokens" 不可行动 — 用户看到后无法采取任何行动。
"上下文 85% ⚠" 可行动 — 用户知道该 `/compact` 或精简对话了。

### P2：渐进展示（Progressive Disclosure）

对话刚开始时，用户不需要关注上下文；对话变长后，上下文成为核心关注点。
信息密度应随对话深度递增，而不是一开始就全量输出。

### P3：安静默认、显式详情（Quiet by Default, Verbose on Demand）

默认只给最小有用信息；详细数据通过斜杠命令按需获取。
避免成为"仪表盘型产品" — 我们是助手，不是监控系统。

### P4：面向未来的扩展性

- 多模型：token 计价不同，统一用%更合理
- 多智能体：子 agent 消耗需要聚合
- 流式进度：实时 token 计数可作为进度指示
- 订阅制 vs API Key：不同计费模式下用户关注点不同

## 三、方案设计

### 3.1 每轮摘要行（Turn Summary）

**核心改变：从 "展示数据" 转向 "传达状态"。**

#### 阶段 A：安静期（上下文 < 50%）

```
─ 1.9s
```

只显示耗时。耗时是唯一在所有阶段都有价值的信息 —
它告诉用户"系统响应快不快"，且不产生认知负担。

#### 阶段 B：感知期（上下文 50%~75%）

```
─ 1.9s · 上下文 62%
```

上下文占比悄然出现。用 dim 色调，不打断思路，但让用户建立
"原来对话越长上下文越满"的心智模型。

#### 阶段 C：警示期（上下文 75%~85%）

```
─ 1.9s · ⚠ 上下文 78%
```

黄色警告图标 + 百分比。用户可以选择 `/compact` 手动压缩，
或继续对话等系统在 85% 自动压缩。

#### 阶段 D：紧急期（上下文 > 85%）

```
─ 1.9s · 🔴 上下文 91%
```

红色指示。若自动压缩已触发，这个状态通常不会持续太久。

#### 为什么不显示原始 token 数？

1. **没有参照系的数字是噪音** — 1,614 tokens 在 200K 窗口中是 0.8%，在 32K 窗口中是 5%，同一个数字含义完全不同
2. **百分比是跨模型通用语言** — 换了模型、换了窗口大小，用户的直觉仍然成立
3. **Claude Code 的产品判断** — 官方明确拒绝了每轮显示 token 的请求，说明他们认为这不是好的 UX
4. **需要详情的用户有 `/usage`** — 不是不给看，是不强制看

### 3.2 斜杠命令体系

#### `/usage` — 当前会话 token 详情

面向需要详细数据的用户（开发者调试、成本敏感用户）。

```
  Token 用量
  ─────────────────────────────
  上下文容量     62%  (31.2K / 50.0K)
  本轮输入       1,614 tokens
  本轮输出       79 tokens
  缓存命中       1,200 tokens
  ─────────────────────────────
  累计输入       12,450 tokens
  累计输出       1,832 tokens
  会话轮次       8 轮
```

设计要点：
- **上下文容量放在最上面** — 最有行动价值的信息优先
- **"本轮"和"累计"区分** — 解决"这个数字是哪一次的"困惑
- **缓存命中单独列** — 让用户感知到系统在帮他省钱

#### `/cost` — 费用估算（仅 API Key 模式）

```
  费用估算
  ─────────────────────────────
  本次会话       $0.032
  输入费用       $0.025  (12.4K tokens × $2.00/M)
  输出费用       $0.007  (1.8K tokens × $4.00/M)
  缓存节省       -$0.005
  ─────────────────────────────
  提示: 使用 /compact 可减少上下文 token，降低后续费用
```

设计要点：
- 仅在 API Key 模式下可用，订阅制用户不需要
- 显示单价信息，帮助用户建立价格直觉
- 底部的 actionable tip 引导用户优化成本

#### `/context` — 上下文可视化

灵感来源：Claude Code 的彩色网格，但更加信息化。

```
  上下文窗口 (50.0K tokens)
  ──────────────────────────────────────────────
  [████████████████████░░░░░░░░░░░░░░░░░░░░░░] 38%

  构成分析:
    系统提示     2.1K  ████
    对话历史    14.8K  ██████████████████████████████
    工具结果     2.3K  █████

  阈值:
    ── 预警 (75%) ──────────── 37.5K
    ── 压缩 (85%) ──────────── 42.5K
    ── 上限 (95%) ──────────── 47.5K

  提示: 上下文 > 85% 时系统将自动压缩，或使用 /compact 手动触发
```

设计要点：
- 进度条一目了然
- **构成分析** — 这是所有竞品都没做的创新：让用户知道上下文被什么占满了
- 阈值可视化 — 用户知道还有多远会触发自动压缩
- 这里可以展示 system prompt 占比（作为构成的一部分，而不是在每轮摘要里）

### 3.3 自动压缩过程的反馈

沿用已实现的 EventBus 事件渲染（`renderCompactStart` / `renderCompactEnd`），
但增强用户可理解性：

```
  ⟳ 上下文压缩中 (LLM 摘要, 42.8K tokens)...
  ✓ 压缩完成: 42.8K → 12.3K (节省 71%)
```

**关键改进：用百分比表达"节省了多少"**，比 `节省 30.5K` 更有感知。

### 3.4 配置维度

通过 `.zhixing/config` 或环境变量控制显示行为：

```yaml
display:
  # 每轮摘要行的详细程度
  # "auto" - 默认渐进式（推荐）
  # "quiet" - 只显示耗时
  # "verbose" - 始终显示完整 token 信息
  turnSummary: "auto"

  # 上下文百分比出现的阈值（auto 模式下）
  contextShowThreshold: 0.5

  # 是否显示费用信息（仅 API Key 模式有效）
  showCost: true
```

**默认就是最佳体验**，配置只服务于有特殊需求的用户。

## 四、信息架构总览

```
用户感知层

  每轮摘要行（被动展示）
  ┌──────────────────────────────────────────┐
  │ 安静 → 感知 → 警示 → 紧急               │
  │ "1.9s" → "1.9s · 62%" → "⚠ 78%" → "🔴" │
  └──────────────────────────────────────────┘
           ↑ 随上下文使用率渐进增强

  斜杠命令（主动获取）
  ┌──────────────────────────────────────────┐
  │ /usage   本轮+累计 token 详情            │
  │ /cost    费用估算（API Key only）         │
  │ /context 容量可视化 + 构成分析           │
  │ /compact 手动触发压缩                    │
  └──────────────────────────────────────────┘

  自动事件（系统驱动）
  ┌──────────────────────────────────────────┐
  │ 压缩进行中 → 压缩完成/失败              │
  │ EventBus 事件 → CLI 渲染                 │
  └──────────────────────────────────────────┘
```

## 五、知行差异化

| 维度 | 竞品做法 | 知行做法 | 为什么更好 |
|------|----------|----------|------------|
| 默认显示 | 关闭或全量 | **渐进式** | 不打扰，又不失察觉 |
| 信息格式 | 原始 token 数 | **百分比 + 状态图标** | 跨模型通用，一看就懂 |
| 上下文构成 | 不展示 | **`/context` 构成分析** | 用户知道什么在占空间 |
| 费用归属 | 总费用 | **输入/输出/缓存分列** | 清楚钱花在哪 |
| 压缩反馈 | 无或简单 | **百分比节省 + 引导** | 用户感知到系统在帮忙 |

## 六、未来演进

### Phase 2：状态栏（TUI 模式）

当知行从 readline REPL 升级到 TUI（Ink/blessed）时，底部状态栏可常驻显示：

```
[知行] model: claude-3.5 | session: 0409-a1b2 | ctx: 38% ████░░░░ | 8 turns
```

这与 OpenClaw TUI 的底栏类似，但信息密度更高。

### Phase 3：多智能体用量聚合

当引入子 Agent 后，`/usage` 需要展示主/子 Agent 的分别消耗：

```
  Token 用量 (含子智能体)
  ─────────────────────────────
  主 Agent        12,450 / 1,832  (入/出)
  └ 代码分析       3,200 / 450
  └ 文件搜索       1,800 / 120
  合计            17,450 / 2,402
```

### Phase 4：流式 Token 计数

在长回复生成过程中，实时显示已生成的 token 数，
作为进度指示（特别是在 reasoning 模型的"思考"阶段）：

```
  ⠸ 思考中... 1.2K tokens
```

### Phase 5：可编程 Status Line

参考 Claude Code 的做法，暴露 JSON 数据供用户自定义脚本渲染：

```json
{
  "context_window": {
    "used_percentage": 38,
    "input_tokens": 19200,
    "remaining_tokens": 30800
  },
  "session": {
    "total_input_tokens": 45000,
    "total_output_tokens": 6200,
    "total_cost_usd": 0.12
  }
}
```

## 七、实现优先级

| 优先级 | 内容 | 依赖 |
|--------|------|------|
| P0 | 每轮摘要行改为渐进式（耗时 → 百分比 → 警告） | ContextEngine 的 usage 数据 |
| P0 | 压缩反馈改为百分比节省 | 已实现 |
| P1 | `/usage` 命令 | TokenEstimator 累计数据 |
| P1 | `/context` 可视化 | TokenEstimator + 构成分析 |
| P2 | `/cost` 命令 | 模型定价配置 |
| P2 | `display` 配置项 | Config 系统 |
| P3 | 状态栏（TUI） | TUI 框架选型 |
| P4 | 多智能体聚合 | 子 Agent 架构 |
| P5 | 可编程 Status Line | 插件系统 |

## 八、与现有架构的集成点

1. **ContextEngine** — 已有 `context:budget_check` 事件，包含 `usageRatio` 和 `status`，是渐进式摘要行的数据源
2. **EventBus** — 已有 `context:compact_start/end` 事件，可直接使用
3. **TokenEstimator** — 需要新增"累计统计"能力（当前只做单次估算）
4. **SessionStore** — 已有 `SessionTokenUsage` 类型，可持久化累计数据
5. **Config 系统** — 需新增 `display` 配置段
